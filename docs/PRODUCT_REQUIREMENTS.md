# ClipMind AI 视频素材管理与智能匹配系统
## 完整需求规格说明书（PRD + 技术架构 + 验收标准）

> 文档版本：V1.0  
> 文档状态：需求评审稿  
> 面向对象：项目负责人、产品负责人、研发、测试、运维、短视频运营、剪辑团队

# 1. 项目背景

在短视频业务启动和扩张过程中，团队会持续积累大量原始视频、拍摄素材、历史成片、产品展示、人物演示、场景镜头和营销素材。传统文件夹管理方式主要依赖文件名、目录结构和人员记忆，随着素材数量增长，容易出现以下问题：

1. 素材越积越多，但无法快速判断某个视频中包含哪些可用镜头。
2. 想寻找特定产品、动作、场景、卖点或镜头类型时，需要逐个打开视频查看。
3. 脚本完成后，无法快速判断现有素材是否能够支撑每一段文案。
4. 已经拍摄过的镜头因为无法被检索，导致重复补拍。
5. 剪辑前需要人工拆分镜头、截图、整理素材、建立清单，准备时间过长。
6. 素材缺少统一标签和质量标准，难以长期沉淀为企业内容资产。
7. AI 自动识别结果缺少人工确认机制，容易误用产品、人物、品牌或风险镜头。

因此，需要建设一套可直接部署到 NAS 的 AI 视频素材管理与智能匹配系统，把分散的视频文件转化为可检索、可理解、可匹配、可复用的镜头资产库。

# 2. 项目目标

## 2.1 总体目标

建设一套面向短视频运营与剪辑团队的 AI 视频素材管理平台，实现：

- 直接读取并索引 NAS 现有素材目录；
- 自动识别视频基础信息；
- 自动拆分可剪辑镜头；
- 自动提取关键帧、缩略图和预览片段；
- 调用 AI API 生成画面描述、标签、质量判断和风险提示；
- 支持自然语言搜索和多条件筛选；
- 支持脚本逐段匹配镜头；
- 输出可直接用于剪辑的镜头清单；
- 支持人工复核、修改和确认 AI 结果；
- 支持片段预览、批量选择、导出和下载；
- 形成可持续增长的企业视频素材资产库。

## 2.2 业务目标

1. 降低寻找素材和浏览视频的时间成本。
2. 提高历史素材复用率，减少重复拍摄。
3. 缩短脚本到剪辑准备的周期。
4. 提高产品、场景、动作和卖点镜头的匹配准确率。
5. 建立统一标签体系和素材质量标准。
6. 为后续自动剪辑、视频复刻、视频生成和内容生产工作流提供素材基础设施。

## 2.3 成功标准

系统上线后应至少达到以下效果：

- 用户无需逐个打开原视频即可理解主要内容；
- 能通过自然语言和筛选条件找到相近镜头；
- 能将完整脚本拆分后推荐候选镜头；
- 能明确标识低置信度和需要人工确认的镜头；
- 能直接下载可用于剪辑的软件片段；
- NAS 新增素材可被增量发现和分析；
- 不移动、不改名、不删除原始视频；
- 系统重启或任务失败后能够恢复和重试。

# 3. 产品定位

ClipMind 不是单纯的“视频文件上传网站”，而是一个面向企业内容团队的：

> 视频镜头资产管理系统 + AI 素材理解系统 + 自然语言检索系统 + 脚本镜头匹配系统。

系统需要同时解决四类问题：

1. **素材管理**：知道素材在哪里、属于什么产品、当前处理到什么状态。
2. **镜头理解**：知道视频中每个镜头出现了什么。
3. **智能检索**：能够根据自然语言、标签和条件快速找到镜头。
4. **脚本匹配**：能够为脚本段落推荐可用画面，并输出剪辑清单。

# 6. 核心业务流程

## 6.1 素材入库流程

```text
NAS 原始素材目录
    ↓
扫描文件与读取元数据
    ↓
计算文件指纹并判断新增/修改/重复
    ↓
生成素材记录
    ↓
进入视频处理任务队列
    ↓
FFmpeg 探测与转码
    ↓
自动拆镜头
    ↓
生成关键帧、缩略图、代理预览和片段
    ↓
调用 AI API 进行画面理解
    ↓
生成描述、标签、产品、质量和风险结果
    ↓
生成文本向量并写入向量索引
    ↓
等待人工审核或直接进入可检索状态
```

## 6.2 自然语言搜索流程

```text
用户输入画面需求
    ↓
系统理解关键词、产品、场景、动作和限制条件
    ↓
生成文本向量
    ↓
向量召回候选镜头
    ↓
结合结构化标签、人工确认状态、风险、质量和时间进行重排
    ↓
返回匹配度、推荐理由、风险提示和候选镜头
    ↓
用户预览、收藏、加入项目或下载
```

## 6.3 脚本匹配流程

```text
用户粘贴脚本
    ↓
AI 自动拆分脚本段落
    ↓
提取每段画面需求、产品、动作、场景、卖点和时长
    ↓
为每段召回多个候选镜头
    ↓
执行去重、时长匹配、相邻镜头差异化和风险过滤
    ↓
生成脚本—镜头匹配表
    ↓
用户人工选择或替换候选镜头
    ↓
导出剪辑清单和片段
```

## 6.4 人工审核流程

```text
AI 分析结果
    ↓
置信度判断
    ├─ 高置信度：自动进入可检索状态
    └─ 低置信度/风险命中：进入待审核队列
            ↓
        人工确认、修改或驳回
            ↓
        保存人工标签和审核记录
            ↓
        更新检索索引
```

# 7. 功能需求

# 7.1 NAS 素材目录管理

## 7.1.1 目录配置

管理员可以添加一个或多个 NAS 素材目录。

每个目录应支持配置：

- 目录名称；
- 物理路径；
- 是否递归扫描子目录；
- 是否启用；
- 允许的文件格式；
- 排除目录；
- 排除文件规则；
- 默认产品；
- 默认业务项目；
- 默认标签；
- 扫描频率；
- 是否自动分析；
- 是否只建立文件索引，不做 AI 分析。

## 7.1.2 原始素材保护

必须遵守：

- 原始素材目录以只读方式挂载；
- 系统不得移动原视频；
- 系统不得修改文件名；
- 系统不得覆盖原视频；
- 系统不得直接删除原视频；
- 删除操作仅删除系统索引和派生文件；
- 如需删除源文件，必须由管理员在 NAS 文件系统中手动完成。

## 7.1.3 增量扫描

系统应识别：

- 新增文件；
- 文件内容变化；
- 文件改名；
- 文件移动；
- 文件删除；
- 同内容不同文件名；
- 同文件名不同内容；
- 重复素材。

建议通过以下组合识别：

- 路径；
- 文件大小；
- 修改时间；
- 快速哈希；
- 完整哈希；
- 视频时长；
- 感知哈希。

## 7.1.4 文件状态

素材状态至少包括：

- 未扫描；
- 已发现；
- 待处理；
- 处理中；
- 已拆镜头；
- AI 分析中；
- 待人工审核；
- 可检索；
- 分析失败；
- 源文件缺失；
- 已暂停；
- 已归档。

# 7.3 视频基础信息分析

系统通过 FFprobe 或等效能力获取：

- 时长；
- 宽度和高度；
- 横竖屏；
- 帧率；
- 视频编码；
- 音频编码；
- 文件大小；
- 比特率；
- 是否有音频；
- 创建时间；
- 旋转信息；
- 色彩空间；
- 关键帧分布；
- 文件可播放性。

异常视频应标记：

- 文件损坏；
- 编码不支持；
- 无法读取；
- 时长为零；
- 音视频不同步；
- 分辨率异常。

# 7.4 AI 自动拆镜头

## 7.4.1 拆分策略

系统需要支持：

1. 基于画面变化的镜头边界检测；
2. 基于固定时长的兜底切分；
3. 基于连续动作的二次切分；
4. 最短镜头合并；
5. 最长镜头强制拆分；
6. 片段首尾安全余量；
7. 黑场和转场识别；
8. 相似连续镜头合并。

## 7.4.2 可配置参数

- 最短镜头时长；
- 最长镜头时长；
- 场景切换阈值；
- 首尾保留时长；
- 是否保留原始音频；
- 是否生成代理预览；
- 代理视频清晰度；
- 是否提前生成可下载片段；
- 是否按需导出片段。

## 7.4.3 镜头信息

每个镜头需保存：

- 镜头 ID；
- 源视频 ID；
- 起始时间；
- 结束时间；
- 时长；
- 镜头序号；
- 关键帧；
- 缩略图；
- 代理预览；
- 原始素材路径；
- 可下载片段路径；
- AI 描述；
- 标签；
- 风险；
- 质量评分；
- 审核状态。

# 7.5 关键帧、缩略图和代理视频

## 7.5.1 关键帧

每个镜头至少生成：

- 主关键帧 1 张；
- 可选辅助关键帧 2～5 张；
- 首帧和尾帧；
- 对长镜头按时间均匀提取关键帧。

## 7.5.2 缩略图

用于列表快速加载，应：

- 采用统一尺寸；
- 保持原比例；
- 支持懒加载；
- 使用 WebP 或 JPEG；
- 体积尽量小；
- 可显示时间码。

## 7.5.3 代理视频

为减少 NAS 带宽压力，生成低码率预览视频：

- 建议 H.264；
- 720P 或更低；
- 保留原比例；
- 保留或关闭音频可配置；
- 支持浏览器直接播放；
- 不替代原始视频。

# 7.6 AI 画面理解

## 7.6.1 输入方式

默认方案：

- 本地使用 FFmpeg 拆镜头；
- 选取每个镜头的关键帧；
- 将关键帧发送给 AI API；
- 必要时发送短时代理视频；
- 不默认上传完整原始视频。

## 7.6.2 输出内容

AI 对每个镜头输出结构化 JSON，至少包含：

- 一句话画面描述；
- 详细画面描述；
- 产品名称；
- 产品型号；
- 产品颜色；
- 产品状态；
- 场景；
- 动作；
- 镜头类型；
- 人物主体；
- 营销用途；
- 卖点；
- 可见文字；
- Logo 或品牌；
- 质量问题；
- 风险项；
- 置信度；
- 是否需要人工确认；
- 搜索关键词；
- 推荐使用场景。

## 7.6.3 AI 输出约束

- 必须为结构化 JSON；
- 字段缺失时返回空值，不得编造；
- 不确定产品时必须标记低置信度；
- 不允许只输出自由文本；
- 模型输出需经过 JSON Schema 校验；
- 校验失败时自动重试；
- 多次失败后进入人工处理队列；
- 原始模型响应需要保留脱敏日志。

# 8. 非功能需求

# 8.1 性能

建议目标：

- 普通页面首次加载不超过 3 秒；
- 搜索结果在 3 秒内返回首屏；
- 单镜头预览在局域网环境下 2 秒内开始播放；
- 支持至少 10 个并发用户；
- 支持至少 100 万个镜头记录的可扩展设计；
- 分页加载，不一次返回全部数据；
- 缩略图和代理视频使用缓存；
- 大任务不得阻塞 Web 请求。

# 8.2 稳定性

- 系统异常不能影响 NAS 原始素材；
- 任务失败可重试；
- 服务重启后任务可恢复；
- 数据库操作具备事务；
- AI API 超时不影响其他任务；
- 网络中断后可继续处理本地任务；
- 磁盘不足时停止生成派生文件并告警；
- 数据库定期备份；
- 系统服务支持自动重启。

# 8.3 安全性

- 原始素材目录只读；
- API Key 不写入前端；
- API Key 加密或使用环境变量；
- 日志不得输出完整密钥；
- 下载需进行权限校验；
- 文件路径必须防止目录穿越；
- 上传文件需校验格式；
- 只允许访问已配置目录；
- 管理操作需记录审计日志；
- 支持 HTTPS 或内网反向代理；
- 默认不开放公网访问。

# 8.4 可维护性

- 支持替换 AI 提供商；
- 配置集中管理；

# 10. NAS 目录规划

```text
/nas/video-assets/                  # 原始素材，只读
├── products/
├── ugc/
├── studio/
├── outdoor/
└── archive/

/nas/clipmind-data/                 # 系统生成数据，可读写
├── database/
├── thumbnails/
├── keyframes/
├── proxies/
├── clips/
├── exports/
├── temp/
├── logs/
├── backups/
└── models-cache/
```

Docker 挂载建议：

```yaml
volumes:
  - /nas/video-assets:/app/source:ro
  - /nas/clipmind-data:/app/data:rw
```

必须保证：

- `/app/source` 为只读；
- `/app/data` 为系统专用；
- 临时目录设置自动清理；
- 数据库和备份目录不与临时文件混放。

# 11. AI 模型与 API 架构

# 11.1 模型适配层

系统不直接把业务代码绑定到某一个模型，应建立统一接口：

```text
AIProvider
├── analyze_frames()
├── analyze_video_clip()
├── generate_embedding()
├── parse_script()
├── rerank_candidates()
└── health_check()
```

支持配置：

- Provider；
- Base URL；
- API Key；
- 模型名称；
- 最大并发；
- 超时时间；
- 重试次数；
- 单次最大图片数量；
- 是否支持视频；
- 是否支持结构化输出；
- 成本参数。

# 11.2 推荐 AI 任务拆分

1. 多模态模型：关键帧理解、产品、动作、场景、质量和风险；
2. 文本向量模型：画面描述和标签向量化；
3. 脚本理解模型：脚本拆分和画面需求提取；
4. 重排模型：候选镜头重排；
5. 可选语音识别模型：提取视频口播内容。

# 11.3 成本控制

系统需记录：

- 每次 AI 调用；
- 模型；
- 输入大小；
- Token 或计费单位；
- 调用耗时；
- 成功或失败；
- 估算成本；
- 素材 ID；
- 镜头 ID；
- 重试次数。

支持：

- 每日额度；
- 每月额度；
- 并发限制；
- 批处理；
- 结果缓存；
- 已分析镜头不重复调用；
- 仅关键帧分析；
- 人工选择后再执行高成本分析。

# 23. 主要风险

## 23.1 NAS 性能不足

风险：

- 视频并发转码会占用大量 CPU；
- 多用户预览影响网络；
- 4K 素材处理慢。

措施：

- 限制 Worker 并发；
- 使用代理视频；
- 夜间批量处理；
- 数据库和缓存使用 SSD；
- 必要时将计算 Worker 放到独立服务器。

## 23.2 AI 识别不稳定

风险：

- 产品识别错误；
- 标签漂移；
- JSON 输出失败；
- 关键帧不足以表示完整动作。

措施：

- 固定 JSON Schema；
- 多关键帧；
- 产品参考图库；
- 低置信度人工审核；
- 结果缓存；
- 模型适配层。

## 23.3 API 成本不可控

措施：

- 仅分析新增镜头；
- 关键帧优先；
- 批量请求；
- 调用记录；
- 每日额度；
- 重复结果缓存；
- 不自动重复分析。

## 23.4 标签体系失控

措施：

- 固定主标签；
- AI 自由描述单独保存；
- 管理员维护标签字典；
- 避免同义标签无限增长；
- 支持标签合并。

## 23.5 原始素材安全

措施：

- 只读挂载；
- 系统数据独立目录；
- 禁止源文件删除接口；
- 文件路径白名单；
- 下载和管理权限分离。

# 26. 结论

ClipMind 的核心价值不是简单地保存视频文件，而是将 NAS 中无法理解、无法检索、难以复用的原始视频，转换成结构化、可搜索、可匹配、可审核、可下载的镜头资产。

本项目建议采用：

- NAS 原始素材只读索引；
- 本地 FFmpeg 处理；
- 外部 AI API 推理；
- PostgreSQL + pgvector；
- Redis + 后台任务；
- Next.js + FastAPI；
- Docker Compose 部署；
- 人工审核兜底；
- 结构化标签与自然语言检索结合；
- 脚本匹配与剪辑清单闭环。

在此方案下，系统既可以较快部署到现有 NAS，也能够在未来继续扩展到自动剪辑、视频生成、视频复刻和完整内容生产工作流。
