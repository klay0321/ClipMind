## 0. 一句话定位（先读这一段）

**当前素材库 = ClipMind 的第一批 seed 样本 + 评测候选种子，仅此而已。**

- 它**只**出现在：示例 / 本地评测说明 / 混淆案例 / 产品识别种子数据（与《业务需求》§8.3、《通用化风险审查》一致）。
- 它**绝不**进入：Python / TypeScript 枚举、SQL CHECK、Alembic 迁移里的固定产品、搜索规则硬编码、
  Prompt 固定产品列表、UI 固定选项、文件夹判断逻辑、排序算法特例。
- 系统管理的是**动态产品目录（Product Catalog）**（见《通用产品目录》），**不是当前文件夹名称列表**；
  当前 seed 的产品数量 / 名称 / 目录结构 / 产品差异**都不得被数据模型、API、UI、识别、检索依赖**。

> **通用化第一原则（贯穿全文）**：新增任意产品都**不需要**改代码 / 加数据库枚举 / 建迁移 / 重设计页面 /
> 改搜索规则 / 改 Prompt 固定列表 / 改文件夹判断 / 加排序特例。当前 seed 只是"先有一批真实数据可以验证管线"，
> **不是**"系统围绕这批数据设计"。两者必须分清，否则就违背了系统的通用化目标。

> 与运营后续动态新增的任意产品**地位完全平等**、**走完全相同的代码路径**。

### 2.2 6 候选 vs 4 产品的口径说明

"产品候选（family×variant）= 6" 是按 `family×variant` 展开的**候选行数**，**不与"4 个已知产品"冲突**：

- 恶魔之眼族 → 软屏 / 硬屏两变体（2 行）；
- 车换挡握把族 → 族级 + 一个待运营确认归并的候选变体别名（2 行）；
- 小键盘族 → 族级 + 一个待运营确认归并的候选变体别名（2 行）。

合计 6 候选行，对应 **4 个已知产品**；其中候选变体别名（如族级与子变体）是**待运营在动态目录中确认归并**的对象，
归并/更名走《通用产品目录》的 `ProductCatalogRevision` 与产品合并/更名机制，**保留旧映射、历史素材关系不丢失**。
**6 与 4 都只是 seed 事实，都不得被任何代码逻辑依赖。**

### 2.3 软屏 vs 硬屏 = 第一组高相似混淆案例（评测门禁，不是特例代码）

- 软屏 / 硬屏是 ClipMind 的**第一组高相似产品混淆案例**，对应《身份血缘》§1.2 的 **ProductConfusionPair**（混淆组）。
- 它在系统里的作用是**产品识别评测的门禁**（见《评测计划》B1 混淆矩阵 / 变体区分阈值命中率），
  **不是**识别流程里的特例代码、**也不是**通用数据结构的硬门禁。
- 通用机制：任意 family 内出现易混变体都用同一套 `ProductConfusionPair` 治理——
  **confusable 变体默认不自动断言**，强制进人工确认，仅当置信高于"变体区分阈值"才给变体级判定（阈值可配，不冻结）。
  软/硬屏只是**第一个被这套通用机制覆盖的实例**，未来任意混淆对（不同品类）**复用同机制、无需改代码**。
- 软硬屏的可见识别特征（屏体柔性 / 包边 / 排线 / 安装方式 / 厚度 / 包装丝印 / SKU 差异）由运营在产品入驻时
  以**动态属性**（`ProductAttributeDefinition` 标 `identity_relevant=true`）+ 参考图角度标注提供（见《通用产品目录》《产品入驻流程》），
  **不写死成软硬屏专用字段**。

## 3. 当前 seed 的合法用途（"能用来验证什么"）

> seed 的价值是**用真实数据把通用管线跑通、并建立可演进的评测基线**，
> 而**不是**证明"系统对公司全部产品的最终准确率"。下列七类用途逐一对齐《评测计划》评测集与《路线图》PR。

### 3.1 产品目录结构验证（通用层级是否成立）

- **验证什么**：用 4 产品 / 6 候选检验 `ProductCategory → ProductFamily → ProductVariant → ProductSKU`
  四级层级、`ProductAlias`（多别名 / 多语言 / 文件夹别名）、`ProductConfusionPair`、`ProductCatalogRevision`
  在**动态创建、无需迁移**前提下能否容纳真实数据，并能否表达"族级 + 待归并变体"这类真实模糊情形。
- **复用 / 新增**：**复用** PR-03B `Product`/`ProductAlias`/`ProductImage`/`AssetProduct`；
  **新增**（落地 PR，非本阶段）family/variant/category 层级与动态属性（见《通用产品目录》、《路线图》PR-A）。
- **红线**：验证只针对**层级结构能否容纳**，**不得**因为 seed 只有 4 产品就把层级简化为"够用就行"。

### 3.2 参考图流程验证（12 角度 + 入驻状态）

- **验证什么**：用 81 张参考图检验 `ProductReferenceAsset` 能否承载 **canonical 参考图角度（12）**——
  正面 / 背面 / 侧面 / 顶部 / 底部 / 接口 / 包装 / 安装前 / 安装后 / 点亮状态 / 关键局部 / 其他——
  以及参考图齐全度如何驱动 **canonical 产品入驻状态（6）**：
  `draft / reference_incomplete / evaluation_required / active / paused / archived`（见《产品入驻流程》）。
- **复用 / 新增**：**复用** `ProductImage`（受控相对路径 `products/{id}/images/`）；**新增**角度枚举值的语义标注与入驻状态机（落地 PR）。
- **红线**：参考图角度是 **12 个 canonical 值**，**不**按 seed 产品（软屏 / 硬屏 / 握把 / 小键盘）定制角度集合。

### 3.3 相似变体混淆验证（混淆组治理是否生效）

- **验证什么**：用软屏 vs 硬屏这**第一组混淆对**检验 `ProductConfusionPair` 治理：
  组内变体**默认不自动断言**、强制人工确认、变体区分阈值高于普通阈值、检索 / 分镜提供"按族 / 按变体"两档。
- **对齐评测**：《评测计划》B1 的**混淆矩阵**与**变体区分阈值命中率**，报告须给**整体**与**混淆组隔离**两套数字，
  混淆组不达标即判产品识别不达标，不被整体高分掩盖。
- **红线**：混淆治理是**通用机制实例化**，软硬屏只是第一个实例；**不得**为软硬屏写专用判定分支。

### 3.4 扫描验证（只读索引 + 稳定身份）

- **验证什么**：用 190 文件 / 8 顶层目录 / "已使用"靠移动（0 字节级重复）检验只读扫描与**稳定 Asset 身份**：
  文件移动到"已使用"子目录后能否按**内容身份**（`quick_hash`/`full_hash`，见《身份血缘》§6）识别为**同一 Asset**，
  而非"缺失 + 新增"两条记录、丢失既有镜头 / 标签 / 使用记录。
- **复用 / 新增**：**复用** PR-01 只读扫描、`ScanRun`、`quick_hash`、ffprobe、缺失检测；
  **新增**（落地 PR，非本阶段）`asset_path_history` 与 content identity 优先匹配策略（见《路线图》PR-C）。
- **红线**：扫描全程只读核对（文件数 / 大小 / mtime），**绝不**移动 / 改名 / 删除源；目录名只表达品类、**不作产品判定真值**。

### 3.5 产品识别基线（开放集识别的起点，不是终点）

- **验证什么**：用 81 参考图 + 视频代表帧（FFmpeg 只读派生关键帧，写 `/app/data`）建立**产品识别基线**，
  检验 **canonical 开放集识别结果（6 态）**能否如实产出：
  `confirmed_product / probable_product / multiple_candidates / unknown_product / no_product_visible / human_review_required`，
  并支持 Top-1 / Top-3 / **未知拒识** / 多产品 / 无产品 / 低置信 / 人工覆盖 / 模型更新后重识别 / **人工结果不被自动覆盖**。
- **对齐评测**：《评测计划》B1（Family/Variant Top-1、Top-3、未知拒识率、错判为已知率、混淆矩阵）。
- **关键限定（避免误读为"准确率证明"）**：
  - seed 规模有限（4 产品 / 6 候选），B1 的数字是**基线 / 回归对照**，**不是**对公司全部产品的最终准确率结论；
  - 真值必须**运营人工确认**（金标准），规则 / AI 预填只进银标准、不定级（见《评测计划》§0）；
  - 识别**绝不**只在已知 4 产品里强选一个——无法识别必须输出 `unknown_product` / 待确认（开放集，见《身份血缘》§1.5）；
  - 识别结果绑定 `model_version` + `catalog_revision`，**模型可替换 / 扩展**，模型更新可重识别，但**人工确认结果不被自动覆盖**。

### 3.6 搜索基线（相关性 / 排序 / 同源 / 未使用优先）

- **验证什么**：用 94 疑似源视频镜头候选 + 运营标注的真实查询建立**搜索基线**，
  检验在现有 Hybrid Search 之上叠加 usage 维度后的相关性、排序、**同源去重**、**未使用优先**表现。
- **对齐评测**：《评测计划》B2（P@5、R@10、NDCG@10、首个可用排名、不相关率、同源重复率、未使用占比）。
- **复用 / 新增**：**复用** PR-04 Hybrid Search、`ShotSearchDocument`（`document_status`/`embedding_status` 正交，
  嵌入不可用仍 `is_searchable`）、E5 embedder + pgvector HNSW；**新增**（落地 PR）usage 过滤 / 降权因子（见《使用感知检索》、《路线图》PR-E）。
- **关键限定**：
  - **未使用 / 已使用**状态真值来自使用血缘（confirmed / legacy 标记），与 B3 共享口径；本库**0 条 confirmed 使用、8 条 legacy 弱证据**，
    因此"未使用优先"基线初期主要由 legacy 负样本与 confirmed 缺失共同构成，**不得**把 legacy 弱证据当作"已使用"参与排序惩罚的硬依据；
  - **降级正确性**：嵌入不可用时搜索基线**仍须可跑**（词法 / 标签 / 结构化召回），指标允许下降但**不得为 0**；
  - 产品过滤来源 = **动态 Product Catalog**，**不是**固定下拉枚举（见《使用感知检索》§7.5）。

### 3.7 动作 / 场景标注模板验证（15 维语义 + 开放动作体系）

- **验证什么**：用 seed 镜头检验 **canonical 通用镜头语义维度（15）**——
  产品 / 产品变体 / 动作 / 场景 / 景别 / 镜头运动 / 拍摄角度 / 人物或手部 / 安装阶段 / 展示阶段 / 包装状态 /
  画幅 / 质量 / 风险 / 营销用途——的标注模板是否落地，重点验证**开放动作体系**：
  基础动作词表 + 公司自定义动作 + 自由文本描述 + 动作别名 + 层级关系，**新动作不需改代码 / 迁移**（见《通用镜头语义》）。
- **对齐评测**：《评测计划》B2 / B4 的动作 / 场景标注模板（`query_labeling_template.csv` / `storyboard_labeling_template.csv`，本地不提交）。
- **红线**：示例动作（展示 / 旋转 / 拿起 / 放下 / 安装 / 拆卸 / 插入 / 按压 / 点亮 / 关闭 / 开箱 / 对比 / 测试 / 局部特写 /
  使用过程 / 完成效果）**仅示例非固定全集**；**不得**把这些动作写成枚举或 Prompt 固定列表。

## 4. 当前 seed **不是什么**（红线，防误用）

> 本节是全文最重要的"否定式边界"。任何把 seed 误当作下列四者之一的做法，都直接违背系统通用化目标，应判不合格。

### 4.1 不是完整产品目录

- 4 产品 / 6 候选 / 8 顶层目录**只是第一批样本**，**不代表**公司当前与未来的完整产品目录。
- 完整产品目录是**运行期动态增长的 Product Catalog**（见《通用产品目录》）；
  公司未来新增任意品类 / 系列 / 变体 / SKU，**不需要**改代码 / 加枚举 / 建迁移 / 改页面 / 改搜索 / 改 Prompt / 改文件夹判断 / 加排序特例。
- **误用示例（禁止）**：把"汽配 / 数码 / 键盘 / 握把"写成 Category 枚举；把"软屏 / 硬屏"写成 Variant 枚举；
  把"恶魔之眼 / 车换挡握把 / 小键盘"写进 SQL CHECK 或迁移固定产品；把当前目录结构写进文件夹判断逻辑。

### 4.2 不是生产训练集

- seed**不**用于训练 / 微调任何识别模型（本阶段也**不接模型、不下载模型权重**）。
- 产品识别是**面向公司产品目录的开放集识别**（识别模型**可替换 / 扩展**），**不是**用 seed 训练出的封闭分类器；
  模型更新可触发重识别，但**人工确认结果不被自动覆盖**（见《身份血缘》§1.5）。
- seed 的角色是**评测候选种子 + 标注预填**，加速运营人工建金标准，**不是**喂给训练循环的标注集。

### 4.3 不是全公司数据集

- ≈4.66 GB / 190 文件**只是被审计到的一个本地目录快照**，**不代表**全公司 NAS 素材的体量、分布、命名习惯或目录组织。
- 数据模型 / API / UI / 识别 / 检索的容量、分页、并发、索引设计**不得**按 seed 规模"够用就行"，
  必须按"公司当前与未来任意规模"设计（与《业务需求》§8 通用化原则一致）。

### 4.4 不是最终准确率证明

- 所有 B1–B4 在 seed 上的数字都是**基线 / 回归对照 / 管线正确性验证**，**不是**对公司全部产品的"最终达标"或"最终准确率"结论。
- **达标判定**必须用**公司真实素材 + 真实 Provider**离线跑、结论人工审阅（见《评测计划》§0 第 5 条、§8）；
  **CI 可用合成 / Fake** 验证指标脚本正确性，但**绝不**以合成数据或 seed 小样本作为"达标"依据。
- seed 规模有限（尤其使用血缘：**0 条 confirmed 使用、0 条可定次数 / 可定成片、本库疑似成片 = 0**），
  B3 成片引用真值**无法从本库直接建立**，必须运营**导入真实成片** + 工程文件解析 + 人工确认才能建金标准。

## 6. seed 数据演进规则（如何从"第一批"走向更多）

> seed 不是一次性的，它会随公司真实数据增长而扩展。演进必须**保持通用化、保持评测可回归、不污染代码**。

1. **新增 seed 产品 = 在动态目录里新增记录，不改代码**：未来加入的任意产品（含新品类）走《产品入驻流程》
   （`draft → reference_incomplete → evaluation_required → active`），与当前 4 产品**同一代码路径**；
   **不**为新 seed 产品建枚举 / 迁移固定项 / UI 固定选项。
2. **seed 与"真实达标数据"分层**：seed 用于建基线 / 管线验证；**达标判定**始终以真实素材 + 真实 Provider 为准；
   seed 扩大不改变"不以合成 / 小样本定级"的原则。
3. **评测集冻结即打版本（vN）**：新增 seed 样本进**下一版本**，**不改历史冻结集**（与"不改历史迁移"同精神，见《评测计划》§2）。
4. **混淆对会增多，机制不变**：软 / 硬屏是**第一组**混淆对；未来任意品类的新混淆对都用同一 `ProductConfusionPair` 机制治理，
   **新增混淆对不需改代码 / 迁移**。
5. **使用血缘 seed 需运营补真实成片**：本库疑似成片 = 0、可定次数 = 0；B3 真值随运营导入真实成片 + 工程文件解析逐步建立，
   `legacy_path_rule` 永远只标"可能用过"、**永不自动计入正式次数**。
6. **产品更名 / 合并不丢历史**：seed 候选变体（如族级 + 待归并变体）归并、或未来产品更名 / 合并，
   走《通用产品目录》`ProductCatalogRevision`，**保留旧映射、历史素材与使用关系不丢失、不需重算为 0**。
