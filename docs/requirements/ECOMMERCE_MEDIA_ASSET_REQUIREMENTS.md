## 0. 公司背景

ClipMind 的真实使用方是一家**跨境电商公司**。公司持续外拍大量**产品视频素材**，原始素材保存在 **NAS**。
剪辑人员从这些素材中挑选镜头，制作**产品带货宣传成片**投放到跨境电商渠道。

ClipMind 的职责是把 NAS 中"无法理解、难检索、难复用"的原始产品视频，转换为
**结构化、可搜索、可匹配、可审核、可复用、可追溯使用历史**的镜头资产，并辅助剪辑人员按
**分镜/脚本**找到最合适、且**未被过度重复使用**的镜头。

> 边界提醒（继承项目硬约束）：ClipMind **只做素材管理与智能匹配**，**绝不做任何生成式视频能力**
> （文生视频/图生视频/数字人/视频复刻/换人换品/自动成片）。所有"关键帧/缩略图/代理/可剪辑片段"
> 一律指 **FFmpeg 从原始视频派生**。源素材**只读**。

### 0.1 本阶段素材库盘点结论（只读审计，事实）

对公司本地素材目录做的**严格只读**审计（脚本 `scripts/discovery/audit_material_library.py`，
明细在 git 忽略的 `.local/material-audit/`，不入库）得到如下事实：

| 维度 | 数值 |
|---|---|
| 文件总数 | 190 |
| 视频 | 102 |
| 图片（产品参考图） | 81 |
| 系统垃圾（Thumbs.db/.DS_Store 等） | 7 |
| 总体积 | ≈ 4.66 GB |
| 顶层目录 | 8（7 个按"日期-品类"拍摄目录 + 1 个 `产品` 参考图目录） |
| 产品候选（family×variant） | 6 |
| 疑似源视频 | 94 |
| 疑似成片 | 0（本库未见明确成片命名） |
| "已使用"证据（位于 `已使用` 目录） | 8 |
| 能据证据确定使用**次数**的 | 0 |
| 能据证据确定对应**成片**的 | 0 |
| 字节级精确重复组 | 0 |
| 疑似低码率/代理视频 | 14 |

> **口径说明（产品候选 6 vs 已知 4 产品）**：6 是按 `family×variant` 展开的**候选行数**——
> 恶魔之眼族含软屏/硬屏两变体（2 行）、车换挡握把族（族级 + 十字架档把变体，2 行）、小键盘族（族级 + mini键盘变体，2 行）；
> 对应 **4 个已知产品**（恶魔之眼软屏、恶魔之眼硬屏、车换挡握把、小键盘），其中"十字架档把""mini键盘"为**待运营确认归并的变体别名**。6 与 4 不冲突。

从盘点直接得到的几条业务事实：

1. **产品参考图与视频分离管理**：`产品/` 目录放参考图（按产品变体分子目录：软屏×2、硬屏、
   十字架档把、mini键盘 等），视频按"日期-品类"目录（如 握把 / 汽配 / 键盘 / 数码 拍摄目录）。
   **视频目录名只表达拍摄品类（汽配/数码/键盘/握把），不直接等于具体产品**。
2. **"已使用"靠移动而非复制**：库内 0 个字节级重复；某汽配拍摄目录的 `已使用/` 子目录中的片段，
   与父目录中的片段是**不同编号的不同片段**——即剪辑人员把"用过的原片**移动**进 `已使用` 子目录"，
   而不是复制。因此**同一素材在库中通常只存一份**，移动后路径变化。
3. **使用次数/成片引用无法从现有目录结构得知**：`已使用` 只能证明"可能曾经用过"，
   既不能确定用过几次，也不能确定对应哪条成片。
4. **软屏 vs 硬屏 极易混淆**：`恶魔之眼` 同时存在软屏、硬屏两个变体，仅凭目录名无法可靠区分，
   需运营提供可见识别特征（见 §产品识别）。

> **⚠️ 样本定位（最高优先级，贯穿全文）**：以上盘点结果**仅来自第一批 seed 样本**，**不代表公司完整产品目录**。
> 当前 4 产品（恶魔之眼软屏/硬屏、车换挡握把、小键盘）与 6 候选**只作 seed 与评测数据**，**绝不进入代码逻辑**
> （不进枚举/SQL CHECK/迁移固定产品/搜索规则/Prompt 固定列表/UI 固定选项/文件夹判断/排序特例）。
> 数据模型 / API / UI / 识别 / 检索 **不得依赖当前产品数量、名称、目录结构或产品差异**。详见 §8 通用化原则与《种子数据范围》。

## 1. 业务问题（公司当前痛点）

> 来自公司直接反馈，逐条对应后续能力设计。括号内标注主要责任能力。

1. **NAS 素材目录混乱**：命名、层级、品类、日期混用，难以浏览与定位。（产品主数据 + 索引）
2. **产品归属不清**：素材属于哪款产品、哪个变体（软/硬屏）不明确。（产品识别）
3. **"已使用"管理依赖目录与后缀**：靠移动到 `已使用` 目录或文件名加"已使用"维护，脆弱且不可统计。（使用血缘）
4. **使用次数不可追踪**：不知道一个素材/镜头被用过多少次。（使用血缘 + 使用记录）
5. **无法知道素材被哪些成片引用**：无法从原镜头反查到引用它的最终成片。（成片反向引用）
6. **搜索结果产品/动作/场景不准确**：检索经常返回产品、动作或场景不相关的素材。（使用感知检索 + 产品硬过滤）
7. **高频素材重复出现**：被反复使用的素材在检索/匹配中仍高频出现，导致成片同质化。（使用次数降权 + 同源去重）
8. **未使用素材得不到优先**：新拍/没用过的素材沉底，浪费拍摄成本。（未使用奖励排序）
9. **分镜计划匹配不稳定**：导入脚本/分镜后，系统不能稳定找到最合适镜头、且常把同一镜头塞进多段。（结构化分镜 + 全局分配）

## 3. 核心对象（领域模型）

> 标注 **[复用]**（现有 ClipMind 模型可直接承载）/ **[扩展]**（在现有上加字段或新增关系）/ **[新增]**（本批新建）。
> 详细字段与映射见 `PRODUCT_IDENTITY_AND_USAGE_LINEAGE_SPEC.md`，差距见 `CURRENT_SYSTEM_GAP_ANALYSIS.md`。
> **产品层级是动态目录（Product Catalog），全部可运营动态创建、无需迁移**——通用层级 `Category → Family → Variant → SKU`
> 与动态属性、入驻流程、版本化见《通用产品目录》《产品入驻流程》。下表的"恶魔之眼/软屏"等仅为 **seed 示例**，非架构依赖。

| 对象 | 状态 | 说明 / 现有承载 |
|---|---|---|
| **Product Category** 产品类别 | [新增] | 顶层品类，**动态创建**。承载未来不同品类产品。详见《通用产品目录》。 |
| **Product Family** 产品族 | [扩展] | 产品线（seed 示例 `恶魔之眼`）。现有 `Product` 为扁平表，需引入 family 分组。 |
| **Product Variant** 产品变体 | [扩展] | 族下可区分版本（seed 示例 `软屏`/`硬屏`）。映射到现有 `Product` 行（每变体一行）。 |
| **Product (SKU)** | [复用] | 现有 `Product`（brand/name/model/sku/status）+ `normalized_name`；SKU 可空。 |
| **Product Attribute** 动态属性 | [新增] | 按 Category 定义的可扩展属性（识别/搜索/运营），**新增属性不需迁移**。详见《通用产品目录》。 |
| **Product Alias** 别名 | [复用] | 现有 `ProductAlias`（含 `normalized_alias`，候选匹配用）。 |
| **Product Reference** 参考图 | [复用] | 现有 `ProductImage`（受控相对路径 `products/{id}/images/`）。 |
| **Source Asset** 源素材 | [复用] | 现有 `Asset`（只读索引，`quick_hash` + `full_hash` 预留、ffprobe 信息、`primary_product_id`、`AssetProduct` 多对多）。 |
| **Shot** 镜头 | [复用] | 现有 `Shot`（generation/时间码/派生关键帧·缩略图·代理）。 |
| **Final Video** 成片 | [新增] | 最终带货成片的索引记录（外部导入，不在 NAS 源目录只读约束内的独立对象）。 |
| **Final Video Usage** 成片使用记录 | [新增] | "某成片在某时间码引用了某源镜头"的引用关系（带证据等级）。 |
| **Storyboard / Segment** 分镜/段 | [复用] | 现有 `ScriptProject`/`ScriptSegment`（结构化需求/产品硬约束/目标时长/锁定）。 |
| **Project** 业务项目 | [复用] | 现有 `Project`（PR-06A，active/archived）。 |
| **Human Review** 人工审核 | [复用] | 现有 `ShotReviewState` + `ReviewEvent` + `ReviewStatus` 状态机（confirm/modify/reject/unable/reopen，乐观锁，stale）。 |

> 关键关系：`Final Video --(Final Video Usage)--> Shot --> Asset --> Product`。
> 正式使用记录的**最小粒度是 Shot**（不是 Asset）；Asset 的使用情况是其所有 Shot 使用的**汇总**。

## 4. 核心流程

```text
外拍产品素材进入 NAS（只读）
  → ClipMind 只读扫描索引（Source Asset）
  → 产品识别（family/variant，文件名/目录规则 + 人工确认 + 后续视觉识别）
  → 拆镜头与内容识别（Shot + 标签：产品/场景/动作/景别…）
  → 人工确认（确认/修改/驳回/无法判断）
  → 搜索与分镜匹配（使用感知检索 + 结构化分镜全局分配）
  → 剪辑人员选镜、锁定、导出剪辑清单
  → 剪辑完成，导入成片或剪辑工程文件（Final Video）
  → 建立"成片 → 原镜头"引用关系（Final Video Usage，带证据等级）
  → 人工审核引用、更新使用次数（仅 confirmed 计入）
  → 下次检索/分镜按使用情况排序（未使用优先、高频降权、同源去重）
```

历史迁移（一次性）：把现有 NAS 的 `已使用` 目录/后缀作为 **legacy 证据**导入，
只标记"可能使用过"，**不自动计入正式使用次数**（见使用血缘规格）。

## 5. 业务状态（使用情况）

> 使用情况按两个**正交轴**描述：**使用频次状态（7 态，canonical 通用集）** 与 **引用确认轴**。
> 全部由 `usage_count`、`last_used_at`、引用确认状态等**派生计算**，阈值**全部配置化、不在此冻结**；
> **对全部产品一致生效（与产品类型无关）**。详见《身份血缘》。

**使用频次状态（7 态，通用，阈值可配）**

| 状态 | 含义 | 主要派生条件（阈值可配） |
|---|---|---|
| **never_used** 从未使用 | 无任何确认引用 | confirmed usage_count = 0 且无 legacy/suspected 证据 |
| **legacy_used_unknown** 历史可能用过 | 仅历史"已使用"弱证据 | 仅 legacy_path_rule，未经人工确认 |
| **used_once** 用过一次 | 被确认使用一次 | confirmed usage_count = 1 |
| **used_multiple** 多次使用 | 被确认多次 | confirmed usage_count ≥ 2（细分阈值可配） |
| **recently_used** 近期使用 | 最近 N 天内确认使用 | last_used_at 在近 N 天（与频次正交，可叠加） |
| **overused** 过度使用 | 高频超复用预算 | confirmed usage_count ≥ 过度阈值（触发降权建议，软信号可人工覆盖） |
| **usage_unknown** 使用情况未知 | 无法定级 | 无可定级证据 / 血缘未建立时的兜底 |

**引用确认轴（正交，关于证据是否人工确认，见《身份血缘》§证据等级与使用次数规则）**：
`usage_pending_review`（有 suspected 引用待确认，不计入正式次数）/ `usage_confirmed`（≥1 条 confirmed 引用，计入）。

> 协调说明：本 7 态替代早期草拟的 8 态混合集——`never_used`≈unused、`used_once`+`used_multiple` 细分原 lightly/heavily、
> `overused` 即原 suggest_deprioritize 的降权信号、`legacy_used_unknown` 显式化 legacy；pending/confirmed 归入正交确认轴。**术语以本 7 态 + 确认轴为准。**

## 6. 非目标（明确不做）

1. **不自动删除或移动 NAS 文件**：ClipMind 永不改动源素材；"已使用"等只在系统索引中标记。
2. **不替代专业剪辑软件**：只做选镜、清单、引用追踪，不做时间线编辑/调色/特效。
3. **不承诺只靠最终 MP4 100% 反查原片**：成片反向引用是**带证据等级与人工确认**的尽力而为，不保证全自动精确。
4. **不在低置信度时强制识别产品**：宁可返回"未知/待确认"，也不伪造产品归属（尤其软/硬屏）。
5. **不自动把疑似引用计入正式次数**：suspected/legacy 证据只提示，不进 `usage_count`；只有 confirmed 计入。
6. **不做任何生成式视频能力**（继承项目硬约束）。

## 7. 验收原则（贯穿后续 PR）

- **真实素材验收**：功能完成必须用**公司真实素材 + 真实 Provider**验收，CI 可用合成/Fake，但不得以合成素材作为"完成"依据。
- **证据分层**：所有自动结论必须标注 `事实 / 规则推断 / AI 推断 / 人工确认`，UI 不伪造"已识别/已匹配/使用次数"。
- **只读安全**：源目录任何流程前后核对文件数/大小/mtime；派生只写 `/app/data`。
- **可回退**：每个能力可关闭/降级而不破坏已有检索（如视觉识别失败回退到名称匹配，嵌入不可用回退到词法/标签召回）。

## 8. 通用化原则（最高架构约束）

> ClipMind 是面向跨境电商公司的**通用产品视频素材管理系统**，必须管理公司**当前与未来任意**产品系列/变体/SKU。
> 当前素材库只是**第一批 seed 样本**；系统**不得围绕当前 4 产品或 6 目录候选硬编码**。

### 8.1 "通用"的准确边界（10 条）

1. 支持公司**不同品类**的产品；
2. 支持**未来新增**产品；
3. 新增产品**不需要修改代码**；
4. 新增产品**不需要增加数据库枚举**；
5. 新增产品**不需要创建数据库迁移**；
6. 新增产品**不需要重新设计页面**；
7. 产品识别模型**允许替换和扩展**；
8. 无法识别时**必须输出 unknown / 待确认**；
9. 当前产品名称**只能作为示例和评测数据，不能进入代码逻辑**；
10. 系统不是面向世界所有物体的通用视觉平台，而是**面向公司产品目录的开放集识别系统**。

> 系统管理的是**动态产品目录（Product Catalog）**，不是当前文件夹名称列表。
> 当前产品名（恶魔之眼软屏/硬屏、车换挡握把、小键盘）只能出现在：示例 / 本地评测说明 / 混淆案例 / 产品识别种子数据；
> **绝不**进入：Python/TypeScript 枚举、SQL CHECK、迁移固定产品、搜索规则硬编码、Prompt 固定产品列表、UI 固定选项、文件夹判断逻辑、排序算法特例。

### 8.2 必须覆盖的产品情形（架构须容纳，不得只适配 seed）

| # | 情形 | 系统应对 |
|---|---|---|
| 1 | 新增完全不同品类的产品 | 动态 Category/Family/Variant，新增免迁移 |
| 2 | 同一 Family 下多个 Variant | Variant 动态创建 |
| 3 | 同一 Variant 下多个 SKU | SKU 动态、可空 |
| 4 | 外观相近但内部规格不同 | 动态属性区分（identity_relevant）+ 混淆组 + 人工确认 |
| 5 | 外观不同但属同一 SKU | 多参考图 / 多角度归于同一 SKU |
| 6 | 包装图与裸机图 | 参考图角度（包装 / 裸机）标注 |
| 7 | 安装前 / 安装后 | 参考图角度 + 镜头语义"安装阶段" |
| 8 | 产品只出现局部 | 局部特征 + 关键局部参考图 + 低置信候选 |
| 9 | 被手 / 包装 / 内饰遮挡 | 遮挡场景 → 低置信候选 / 人工确认 |
| 10 | 视频中多个产品 | 多产品识别（multiple_candidates） |
| 11 | 视频中无产品 | `no_product_visible` |
| 12 | 未知产品 | `unknown_product` + 人工确认，**不强制猜测** |
| 13 | 已停用产品 | 入驻状态 `paused` / `archived`，历史关系保留 |
| 14 | 产品更名 | 保留旧映射，**历史素材关系不丢失** |
| 15 | 产品合并 | 保留旧产品映射，引用不丢失 |
| 16 | SKU 迁移 | Catalog Revision 记录，引用随迁不丢 |
| 17 | 多语言商品名 | 多语言名称 + 别名 |
| 18 | 文件夹名 ≠ 正式产品名 | 文件夹别名只作候选线索，**不作判定真值** |

### 8.3 当前产品的唯一合法用途

当前 4 产品 + 6 候选标为 **seed dataset only**：用于产品目录结构验证、参考图流程验证、相似变体（软/硬屏）混淆验证、扫描验证、产品识别基线、搜索基线、动作/场景标注模板验证。详见《种子数据范围》《通用化风险审查》。
软屏 vs 硬屏 = **第一组高相似产品混淆案例**，是产品识别评测的门禁，**不是**识别流程的特例代码、也不是通用数据结构的硬门禁。

## 9. 已锁定的第一版业务决策（Phase 0 / 0.5 收口，权威）

> 本节是经方向审查通过、**已锁定**的第一版通用业务决策，作为后续所有 PR 的约束源；详细规格见《通用产品目录》《产品入驻流程》《身份血缘》《通用化风险审查》《路线图》，三者须与本节一致。

1. ClipMind 面向公司**全部当前与未来产品**，**不为当前四款产品定制**。
2. 当前素材库只是 **seed dataset**，**不是**完整公司产品目录。
3. 产品层级为 **Category → Family → Variant → SKU**。
4. **Family 是核心产品实体**；**Category 建议必填**；**Variant 与 SKU 可选**。
5. 新产品后续**只新增数据**，**不修改**代码、枚举、SQL CHECK 或数据库结构。
6. **PR-A 初次建立产品目录基础表时，允许且必须使用正式 Alembic migration**（一次性建基础 Schema）；**此后新增产品/属性数据不需要迁移**。
7. **中文正式名称必填**；英文名称可选；**Alias 支持多语言名称与文件夹别名**。
8. 新产品先进入 **Draft**，确认后才能 **Active**。
9. 参考图允许**从 1 张开始**；**3 张为建议最低**；**5 张以上适合后续视觉识别**。
10. 产品识别不确定时返回 **`unknown_product` + `human_review_required`**，**不强制猜测**。
11. **产品更名不改变稳定 ID**。
12. **产品合并保留旧 ID、历史素材、使用关系与重定向记录**。
13. 动态属性必须由**受类型约束的定义驱动**，**不允许**完全自由且不可验证的 JSON。
14. 当前 **seed 产品名称不得进入生产代码、SQL、固定 Prompt 或 UI 枚举**。
15. **自动产品识别、使用次数、使用血缘、检索重排不属于 PR-A1**（见《路线图》PR 拆分）。

> 关键区分（务必贯穿）：**“PR-A1 一次性建立通用基础 Schema（正式迁移）”** 与 **“此后新增产品/属性是纯数据、免迁移”** 是两回事，不得混为“PR-A 不建迁移”。
