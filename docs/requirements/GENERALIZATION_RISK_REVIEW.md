## 0. 审查方法与最高原则

### 0.1 最高原则（贯穿全文）

新增公司**任意**产品（当前与未来任意品类 / 系列 / 变体 / SKU）**都不需要**：
改代码 / 加数据库枚举 / 建数据库迁移 / 重设计页面 / 改搜索规则 / 改 Prompt 固定列表 / 改文件夹判断 / 加排序特例。

- 系统管理的是**动态产品目录（Product Catalog）**（canonical 层级 `ProductCategory → ProductFamily → ProductVariant → ProductSKU`，
  全部可运营动态创建、无需迁移），**不是当前文件夹名称列表**。
- 产品识别是**面向公司产品目录的开放集识别**（canonical 6 态：`confirmed_product` / `probable_product` /
  `multiple_candidates` / `unknown_product` / `no_product_visible` / `human_review_required`），
  不是世界通用视觉平台；无法识别**必须输出 `unknown_product` / 待确认，绝不强制猜测**。
- 识别模型**允许替换 / 扩展**；模型或目录版本更新可触发**重识别**，但**人工确认结果不被自动覆盖**。

### 0.2 当前 seed 产品的唯一合法边界

当前已知 4 产品（**恶魔之眼软屏 / 恶魔之眼硬屏 / 车换挡握把 / 小键盘**；其中恶魔之眼软 / 硬屏 = **第一组高相似混淆案例**）
与盘点出的 6 个 `family×variant` 候选，**只能**作为：示例 / 本地评测说明 / 混淆案例 / 产品识别种子数据。
**绝不**进入：Python / TypeScript 枚举、SQL CHECK、迁移里的固定产品、搜索规则硬编码、Prompt 固定产品列表、
UI 固定选项、文件夹判断逻辑、排序算法特例。当前 190 文件 / 102 视频 / 81 参考图 / 6 候选盘点结果 = **第一批 seed 样本**，
**不代表公司完整产品目录**；数据模型 / API / UI / 识别 / 检索**不得依赖当前产品数量 / 名称 / 目录结构 / 差异**（见《种子数据范围》）。

### 0.3 硬编码审计结论（本文件审查的事实基线）

对当前仓库代码做的只读硬编码审计（搜索 seed 产品名与品类词在 `*.py` / `*.ts` / `*.tsx` / `*.sql` 中的出现）结论：

- **seed 产品名（恶魔之眼 / 软屏 / 硬屏 / 车换挡握把 / 小键盘 / 十字架档把 / mini键盘）与品类词（汽配 / 数码 / 键盘 / 握把）
  仅出现在 `scripts/discovery/audit_material_library.py` 及其测试**——这是一次性**只读审计工具**，用于生成 seed 候选与
  空真值标注模板（属《种子数据范围》允许的"种子 / 评测数据"唯一合法用途），**不参与任何运行时业务逻辑**。
- 在**运行时应用代码**（`packages/shared/clipmind_shared/models/`、API service、前端、Alembic 迁移、SQL CHECK）中
  **未发现任何 seed 产品名 / 品类词硬编码**。现有 `Product` 模型为 `brand/name/model/sku/normalized_name/status` 的
  **数据驱动**结构（`models/product.py`），产品全部是**数据行**而非代码常量。
- 现有枚举（`models/enums.py`）中 `ProductStatus` 仅 `active/archived`，**无任何具体产品作为枚举值**；
  其余枚举均为状态机 / 维度类型（如 `TagType=product/scene/action/...`），**不含产品身份**。

> 该结论是好消息：ClipMind 现有主干**本就数据驱动**，通用化风险的主战场在**未来落地 PR 不要引入新硬编码**，
> 而非"清理现有硬编码"。下表 20 项中多数"当前状态"为**架构上已规避**，PR-A 主要是**守住边界 + 补层级 / 入驻 / 混淆组治理**。

### 0.4 表头约定

每项审查统一给出五栏：

- **风险**：过拟合发生时的具体坏后果；
- **当前状态**：现有代码 / 模型 / 文档的事实（含硬编码审计结论，标【已规避】/【部分】/【待建】）；
- **架构措施**：如何从结构上根除该风险，明确**复用现有 X** / **新增 Y**；
- **测试方式**：如何用可执行测试 / 评测守门（CI 可 Fake/合成，**功能达标须真实素材 + 真实 Provider**，见《评测计划》）；
- **PR-A 是否需处理**：是 / 否 / 部分（**PR-A 已拆为 PR-A1 目录核心 + PR-A2 动态属性 / 参考图 / 入驻 / 混淆组**，见《路线图》；下文"PR-A"泛指这两个落地 PR——目录层级/更名/合并/别名属 PR-A1，属性/参考图/混淆组/入驻属 PR-A2）。

## 1. 逐项风险审查（20 项）

### 风险 1：产品名写死（在代码 / SQL / Prompt 里直接出现"恶魔之眼"等）

- **风险**：任一 seed 产品名进入运行时代码、SQL、迁移或 Prompt，则新增产品需改代码 / 重部署，且改名即失效。
- **当前状态**：【已规避】。硬编码审计：seed 产品名**仅在** `scripts/discovery/audit_material_library.py`（只读审计工具，生成种子 / 模板）出现，
  运行时应用代码、模型、迁移、SQL CHECK 中**零出现**。`Product` 是数据行（`models/product.py`），产品名是 `Product.name` 列值。
- **架构措施**：**复用** 现有 `Product.name` / `normalized_name` / `ProductAlias`（数据驱动，命中靠运行期目录数据）；
  **新增**《通用产品目录》的 `ProductCategory/ProductFamily/ProductVariant/ProductSKU/ProductAlias` 层级，产品名一律是**数据**。
  约束：审计脚本中的 seed 名是**评测种子用途**（《种子数据范围》允许），**不得**被任何业务代码 import。
- **测试方式**：CI 加**反硬编码门禁测试**：grep 运行时代码目录（`packages/`、`apps/`、`migrations/`，排除 `scripts/discovery/`）
  断言不含 seed 产品名集合；产品 CRUD 测试用临时虚构名（如"测试产品X"）验证全流程不依赖具体名。
- **PR-A 是否需处理**：**是**（守门 + 把扁平 `Product` 升级为层级时，确保新名全是数据行；加反硬编码 CI 门禁）。

### 风险 2：产品数量写死（假设"就 4 个产品 / 6 候选"）

- **风险**：任何 `len==4` / 固定 6 行 / 固定分页大小 / 固定下拉项数等假设，导致第 5 个产品起出错或被截断。
- **当前状态**：【已规避】。现有产品为不定行数据表，无"产品数量"常量；检索 / 列表均分页（`paginate`，见《使用感知检索》§1.1 复用稳定排序与分页）。
- **架构措施**：**复用** 现有分页 / 游标与候选池有界机制；**新增** 目录层级同样不限行数。所有"产品集合"一律按数据查询，
  禁止任何基于 seed 计数（4 / 6）的常量或测试断言。
- **测试方式**：构造 **N≫6**（如 50）个虚构产品的 fixture，验证目录 CRUD、检索产品过滤、UI 选项生成、识别候选不因数量变化退化或截断；
  断言无任何 `== 4` / `== 6` 数量假设。
- **PR-A 是否需处理**：**是**（层级表建好后用大 N fixture 回归，确保数量无关）。

### 风险 3：Product Enum 写死（把产品做成 Python/TS 枚举或 SQL CHECK）

- **风险**：产品身份一旦进枚举 / CHECK，新增产品必须改枚举 + 写迁移 + 重部署，彻底违反通用化。
- **当前状态**：【已规避】。`models/enums.py` 审查确认：**无任何产品身份枚举**；`ProductStatus` 仅 `active/archived`（状态机，非产品）。
  产品是表数据，PG enum 只用于**状态 / 维度**（状态机值集稳定有限），不用于产品。
- **架构措施**：**复用** 现有"产品=数据表、enum 仅状态机"的既定分工；**新增** 层级 / 属性 / 别名全部是**数据行**，
  动态属性走《通用产品目录》的 `ProductAttributeDefinition`（受定义约束的可扩展属性，新增属性免迁移），
  **既不把产品写成 enum，也不退化为无约束 JSON**。约束：禁止为"产品类型"新建任何 enum / CHECK。
- **测试方式**：迁移 / 模型审查清单（人工 + CI 静态扫描）断言无产品类 enum / CHECK；新增一个虚构 Category + Family + Variant
  的集成测试**不触发任何迁移**即可完成。
- **PR-A 是否需处理**：**是**（PR-A 新建 `product_family` 等表时，明令产品身份不入 enum / CHECK；属性体系用定义表而非枚举）。

### 风险 4：UI 选项写死（前端产品下拉 / 筛选器固定列出 seed 产品）

- **风险**：前端把产品选项写成静态数组，新增产品前端不显示，需改前端代码重新发版。
- **当前状态**：【部分 / 待建】。当前前端尚无"通用产品目录选择器"（产品识别 / 目录 UI 属后续 PR）；现有筛选已倾向由后端数据驱动。
  《使用感知检索》§7.5 已冻结："产品过滤来源 = Product Catalog（运行期数据），UI 的产品选项**由目录动态生成**，新增产品无需改前端或后端代码"。
- **架构措施**：**复用** 现有 TanStack Query + 后端列表 API 的数据驱动渲染模式；**新增** 目录 API（family/variant/category/SKU/属性列表）
  供前端**动态渲染**选项；UI 不得内置任何 seed 产品常量数组。证据分层：UI 不伪造"已识别 / 已匹配 / 使用次数"。
- **测试方式**：前端组件测试用 mock 目录 API 返回**虚构产品集**，断言下拉 / 筛选项**完全来自接口数据**、无静态产品字面量；
  e2e（隔离 Fake）验证新增产品后前端无需改码即出现。
- **PR-A 是否需处理**：**部分**（PR-A 产出 family/variant 目录 API；产品库管理页扩展见《路线图》PR-A UI，识别审核 UI 在 PR-D/PR-F）。

### 风险 5：Prompt 产品列表写死（识别 / 解析 Prompt 内嵌固定产品清单）

- **风险**：把 seed 产品清单塞进识别 / 查询解析 Prompt，则模型只认这几款，新增产品识别不到、改 Prompt 才行，且诱导"强行从已知里选"。
- **当前状态**：【部分 / 待建】。视觉产品识别为后续 PR（《现状差距分析》§1.13、《路线图》PR-F，本阶段不接模型）。
  查询解析已接（`MiMoQueryParser`，失败降级 rulebased），但不内嵌产品固定清单。开放集识别 6 态已在《身份血缘》§1.5 冻结。
- **架构措施**：**复用** 现有 Provider 抽象 + 占位降级（`NotConfiguredProvider`，绝不伪造）；**新增** 识别 Prompt 设计原则——
  候选产品**从运行期 Product Catalog 注入**（带 `catalog_revision`），**不在 Prompt 写死清单**；Prompt 必须允许输出
  `unknown_product` / `multiple_candidates` / `no_product_visible`，**绝不强制在已知产品里选一个**；confusable 变体默认不自动断言。
- **测试方式**：识别 Provider 契约测试断言"候选集来自传入目录参数、非 Prompt 常量"；喂入**目录外**画面必须返回 `unknown_product`
  而非强选（《评测计划》B1 未知拒识率 / 错判为已知率）；模型替换后重识别但人工确认不被覆盖的回归测试。
- **PR-A 是否需处理**：**否（PR-A 不接模型）**；PR-A 仅冻结目录与混淆组供后续注入。识别 Prompt 落地见《路线图》PR-F（强制人工确认）。

### 风险 6：文件夹结构写死（用目录名直接判定产品 / 把"汽配/数码"当产品）

- **风险**：把拍摄目录名（汽配 / 数码 / 键盘 / 握把）当作产品真值，则目录改名 / 新目录即错判，且目录名 ≠ 正式产品名（审计事实）。
- **当前状态**：【已规避】。审计明确"视频目录名只表达拍摄品类，不直接等于具体产品"（《业务需求》§0.1）；
  品类词仅在审计脚本作**候选线索**。《身份血缘》§1.4 冻结"**绝不只根据目录名最终确认产品身份**"，目录命中 = `规则推断`（仅候选 needs_human）。
- **架构措施**：**复用** 现有 `Asset` 路径 / 扫描机制（路径只作元数据）；**新增** 目录别名只作**候选线索**（写入 `ProductAlias` 文件夹别名 / 候选证据），
  **不作判定真值**；识别证据分层（事实 / 规则推断 / AI 推断 / 人工确认），目录规则命中只产候选。稳定身份按内容（content identity）而非路径（《身份血缘》§6）。
- **测试方式**：构造"目录名与真实产品不一致"样本，断言系统只产候选 needs_human、不自动确认；目录改名后产品关系不丢（呼应风险 17）。
  《评测计划》B1 明令"目录品类词不得作真值"。
- **PR-A 是否需处理**：**部分**（PR-A 支持 family 级"文件夹别名"作候选线索；判定真值靠人工确认 / 后续视觉识别）。

### 风险 7：中文关键词写死（搜索 / 匹配规则内嵌中文产品 / 动作关键词表）

- **风险**：检索 / 解析里硬编码中文关键词（产品词、动作词），则换产品 / 换说法 / 多语言即失配，新增需改规则。
- **当前状态**：【部分】。现有 Hybrid Search 是词法（pg_trgm）+ 语义 + 标签 + 产品 + 结构化召回（数据驱动，《现状差距分析》§1.5），
  不依赖固定产品关键词表；查询解析有 rulebased 兜底但产品来自目录解析。动作体系拟为"基础动作词表 + 公司自定义 + 自由文本 + 别名 + 层级"（《通用镜头语义》），非固定全集。
- **架构措施**：**复用** 现有多通道召回 + `normalized_*` 归一化匹配（不靠脆弱文本规则）；**新增** 动作 / 属性词表全部**数据驱动可扩展**
  （新动作免改码 / 免迁移）；多语言走 `ProductAlias` 多语言名称。检索产品过滤来源 = 动态 Catalog（《使用感知检索》§7.5），不写死关键词。
- **测试方式**：用虚构产品 + 虚构动作词验证检索 / 解析全流程；多语言别名命中测试；断言无内嵌 seed 中文产品 / 动作常量表（反硬编码门禁覆盖）。
- **PR-A 是否需处理**：**否**（检索 / 动作扩展属 PR-E / 通用镜头语义；PR-A 仅保证产品别名数据驱动）。

### 风险 8：产品属性写死（把"屏幕软硬 / 接口类型"等做成固定列或固定 JSON key）

- **风险**：不同品类属性差异大；若写死列或无约束 JSON，则要么新增属性需迁移、要么属性无类型 / 无法搜索 / 无版本。
- **当前状态**：【待建】。现有 `Product.selling_points`(JSONB) 是松散卖点，无类型化属性体系；动态属性是新增需求（《业务需求》§3 Product Attribute [新增]）。
- **架构措施**：**新增**《通用产品目录》的**动态属性体系**：`ProductAttributeDefinition`（`category_id, key, display_name, value_type∈{text,number,boolean,enum,multi_enum,measurement,date}, allowed_values, required, searchable, identity_relevant, version`）
  + `ProductAttributeValue`。**稳定核心 + 受定义约束的可扩展属性**：既不全写死列、也不无约束 JSON；不同 Category 可定义不同属性，
  新增属性免迁移、属性变化保留版本；明确标注哪些 `identity_relevant`（参与识别）/ `searchable`（参与搜索）/ 仅运营信息。
  软 / 硬屏的"可见识别特征"即落为 `identity_relevant=true` 的动态属性 + confusable 组（《身份血缘》§1.2/§1.3）。
- **测试方式**：为虚构 Category 定义新属性（含 enum/measurement）并赋值、检索、参与识别，全程**不触发迁移**；
  属性改版保留旧版本测试；`searchable` 属性进查询、`identity_relevant` 属性进识别证据的契约测试。
- **PR-A 是否需处理**：**部分**（属性体系主体在《通用产品目录》对应落地；PR-A 至少落 confusable 组与 family/variant 的可见特征字段，
  属性定义表可与 PR-A 同期或紧随，按《路线图》评审，但 seed 软/硬屏特征必须以**动态属性**而非固定列承载）。

### 风险 9：只支持单产品画面（假设一帧 / 一镜头只有一个产品）

- **风险**：若数据模型 / 识别假设单产品，则多产品同框、对比镜头无法表达，违反真实带货素材。
- **当前状态**：【已规避】。现有 `AssetProduct` 是**多对多**（一个素材可含多个产品，`asset.primary_product_id` 仅"主产品"，见 `models/product.py` 注释）；
  `ShotTag(type=product)` 可多条。识别 6 态含 `multiple_candidates`（《身份血缘》§1.5）。
- **架构措施**：**复用** 现有 `AssetProduct` 多对多 + 多 `ShotTag`；**新增** 识别结果 `top_candidates` 数组 + `multiple_candidates` 态，
  支持一镜头多产品、对比镜头；使用记录粒度是 Shot，可挂多个产品归属。
- **测试方式**：构造多产品同框样本，断言可关联多个产品、识别返回 `multiple_candidates` 且进人工确认；检索"含产品A且含产品B"可命中。
- **PR-A 是否需处理**：**否（已具备）**；PR-A 不破坏多对多。识别多候选落地在 PR-F。

### 风险 10：强制识别已知产品（在已知集合里硬选一个，不会拒识）

- **风险**：开放集退化为闭集分类，目录外 / 遮挡 / 无产品画面被强行贴上某 seed 产品，污染数据、误导剪辑。
- **当前状态**：【已规避（规格层）/ 待落地】。《身份血缘》§1.5 已冻结开放集 6 态、`unknown_score`、"绝不强制猜测"；本阶段不接模型故无实现可越界。
- **架构措施**：**新增** 开放集识别契约：识别结果必含 `unknown_score` 与 6 态判别；低置信 / 多产品 / 遮挡 / 无产品 →
  `human_review_required` / `unknown_product` / `multiple_candidates` / `no_product_visible`，**绝不在已知集合里强选**；
  confusable 变体仅当置信 ≥ 变体区分阈值才自动判定，否则进人工确认。**复用** 现有 `ShotReviewState` 承接待确认。
- **测试方式**：《评测计划》B1 的**未知拒识率**与**错判为已知率（false-known rate）**专项；故意混入目录外 / 无产品画面，断言拒识而非强选。
- **PR-A 是否需处理**：**部分**（PR-A 建混淆组 + 变体区分阈值机制，定义"何时不自动判定"；识别实现与拒识跑分在 PR-F + B1）。

### 风险 11：没有 unknown（系统无"未知 / 待确认"出口）

- **风险**：缺 `unknown_product` 出口，则识别要么强选、要么丢弃，运营无法处理新产品 / 边界画面，数据失真。
- **当前状态**：【已规避（规格层）】。6 态含 `unknown_product`，使用频次 7 态含 `usage_unknown`、`legacy_used_unknown`（《业务需求》§5、《身份血缘》§4），
  审核状态含 `unable`（无法判断，`ReviewStatus.UNABLE`，见 `enums.py`）。"未知 / 待确认"是一等公民。
- **架构措施**：**复用** 现有 `ReviewStatus.UNABLE` + `ShotReviewState`；**新增** 识别 `unknown_product` / 使用 `usage_unknown` / `legacy_used_unknown`
  作为显式状态贯穿识别 / 使用 / 检索；UI 必须可展示"未知 / 待确认 / 使用情况未知"，不伪造结论。
- **测试方式**：识别 / 使用 / 检索三处各有"unknown 可表达、可过滤、可在 UI 标注"的测试；断言 unknown 不被当作某确定产品 / 某确定次数。
- **PR-A 是否需处理**：**部分**（PR-A 让产品识别结果可落 `unknown_product`；使用类 unknown 在 PR-B；UI 在 PR-D/PR-F）。

### 风险 12：训练 / 测试集泄漏（评测真值用了系统自己产的预填，或同集既调参又验收）

- **风险**：用规则 / AI 预填当真值，或同一冻结集既调权重又报达标，指标虚高，无法反映真实泛化。
- **当前状态**：【已规避（评测规格层）】。《评测计划》§0 冻结："不自动伪造人工真值""规则 / AI 预填进银标准、只加速标注不定级""只有人工确认进金标准"；
  评测集冻结打版本、新增进下一版、不改历史集。
- **架构措施**：**新增**（评测域，不改业务表）金 / 银标准分离、IAA 横切集（B0）先于定级、评测集版本化；
  权重 / 阈值在落地 PR 调，**达标判定用独立冻结集 + 真实素材 + 真实 Provider**，CI 仅验管线正确性（合成不作达标依据）。
- **测试方式**：评测脚本断言金标准只含 `人工确认` 来源；调参集与达标集分离的流程检查；B0 一致性达标后该维度才定级。
- **PR-A 是否需处理**：**否（属评测体系）**；PR-A 仅建 B1 类目体系（产品身份），评测纪律由《评测计划》统一约束。

### 风险 13：参考图来自同一视频（产品参考图与被测镜头同源，识别"作弊"）

- **风险**：若 family/variant 的参考图与评测 / 被识别镜头出自同一视频 / 同一帧，识别命中是"记住了"而非泛化，真实新素材会失败。
- **当前状态**：【部分 / 待规范】。审计：参考图（81 张）放独立 `产品/` 目录、与视频按"日期-品类"目录分离（《业务需求》§0.1），物理上倾向分离；
  但"参考图与被测镜头不同源"尚未作为评测硬约束显式断言。
- **架构措施**：**复用** 现有 `ProductImage`（参考图独立受控路径 `products/{id}/images/`）与 content identity（`quick_hash/full_hash`，《身份血缘》§6）；
  **新增** 评测构建约束：参考图与 B1 被测样本必须**内容身份不同源**（按 full_hash/quick_hash 判定，非路径），同源样本剔除或单列。
- **测试方式**：评测建集脚本对"参考图 vs 被测帧"做 content identity 去重断言；报告标注"同源样本数=0"作为 B1 有效性前置（《评测计划》B1）。
- **PR-A 是否需处理**：**部分**（PR-A 建参考图与类目体系；同源剔除规则属评测脚本，但 PR-A 须保证参考图有稳定内容身份可供去重）。

### 风险 14：同拍摄批次泄漏（同一天 / 同一布光批次的素材分散进训练与测试，泛化被高估）

- **风险**：同批次（同日 / 同机位 / 同布光 / 同背景）素材高度相似，若跨集分散，指标高估，换批次 / 换季拍摄即跌。
- **当前状态**：【待规范】。审计顶层是"日期-品类"目录（隐含批次信息），但批次未作为评测划分维度显式约束。
- **架构措施**：**新增** 评测划分原则：按**批次（拍摄日 / 目录 / 机位）分组留出**（group-wise split），同批次不跨"调参 vs 验收"边界；
  报告记录批次分布。**复用** 现有路径 / 扫描元数据（拍摄目录可作批次代理，仅用于评测划分，不作产品真值）。
- **测试方式**：评测划分脚本断言"同批次组不同时落入调参集与达标集"；对比"随机划分 vs 批次留出"指标差，量化泄漏风险（《评测计划》B0/B1）。
- **PR-A 是否需处理**：**否（属评测划分纪律）**；PR-A 不涉及，归《评测计划》落地。

### 风险 15：当前四款好但新产品失败（系统在 seed 上调到很好，泛化到第 5 款崩）

- **风险**：所有阈值 / 权重 / 特征围绕 4 款 seed 调优，过拟合 seed；新品类 / 新外观产品识别 / 检索 / 匹配全面退化。
- **当前状态**：【架构规避 / 持续风险】。架构上一切数据驱动、阈值可配（《使用感知检索》《结构化分镜匹配》均"不冻结权重"），
  但"只在 4 款上验收"本身就是过拟合来源——这是**纪律风险**而非结构缺陷。
- **架构措施**：**复用** 现有可配置因子 / 可降级回退（视觉失败回退名称 / 目录规则、嵌入失败回退非向量召回、全局分配失败回退贪心）；
  **新增** 评测纪律：识别 / 检索 / 匹配的达标须含**未知样本 + 混淆组 + 跨批次**，并以"对新产品可降级到人工 + 候选"为可接受兜底（而非要求新品自动全对）。
  关键：**新增产品不需开发介入即可被系统容纳与人工兜底**（见风险 16）。
- **测试方式**：用**完全虚构的新品类产品**跑端到端冒烟（目录创建 → 参考图 → 扫描关联 → 检索过滤 → 分镜产品硬过滤 → 识别返回候选 / unknown），
  断言"功能可用、低置信走人工"而非崩溃；《评测计划》各集含未知 / 混淆 / 跨批次专项。
- **PR-A 是否需处理**：**部分**（PR-A 保证新产品目录 / 别名 / 混淆组可被纯数据创建并被检索 / 匹配消费；模型泛化跑分在 PR-F/B1）。

### 风险 16：新产品需开发介入（运营无法自助新增产品，必须找工程师改码 / 迁移）

- **风险**：若新增产品要工程师改代码 / 写迁移 / 发版，则违反"通用化最高原则"，公司扩品类即受阻。
- **当前状态**：【部分 / 待建】。现有产品 CRUD 已数据驱动（可增 `Product` 行），但层级（category/family/variant）、动态属性、入驻流程尚未建（新增需求）。
- **架构措施**：**复用** 现有产品 / 别名 / 参考图 CRUD（PR-03B）；**新增**《通用产品目录》全层级 + 动态属性 + 《产品入驻流程》6 态
  （`draft / reference_incomplete / evaluation_required / active / paused / archived`）与 12 角度参考图体系，
  使运营**纯通过 UI / API 数据操作**完成新增（创建 Category/Family/Variant/SKU、传参考图、定属性、标混淆组），**无需任何迁移 / 改码 / 发版**。
- **测试方式**：集成测试模拟运营全自助入驻一个虚构新品类产品到 `active`，**全程零迁移、零代码改动**；断言入驻状态机流转正确、参考图角度齐备校验生效。
- **PR-A 是否需处理**：**是**（PR-A 是"运营可自助维护产品台账"的第一步：family/variant/alias/混淆组 CRUD；
  完整入驻 6 态与属性体系按《产品入驻流程》《通用产品目录》落地，PR-A 打基座）。

### 风险 17：产品更名失效（产品改名后历史素材 / 镜头 / 使用关系断裂）

- **风险**：若产品身份靠名字字符串关联，改名即丢失既有素材关系、使用记录、检索命中。
- **当前状态**：【已规避（结构层）】。现有关联靠**外键 id**（`AssetProduct.product_id`、`ShotReviewState.confirmed_product_id`），
  非名字字符串，改 `Product.name` 不断关系。《业务需求》§8.2(14)、《身份血缘》§2.1 已冻结"更名保留旧映射、历史素材关系不丢失"。
- **架构措施**：**复用** 现有 id 外键关联（改名只改显示名，关系不动）；**新增** `ProductAlias` 保留旧名为别名（旧名仍可检索命中）+
  《通用产品目录》`ProductCatalogRevision` 记录更名修订（审计可追溯）；多语言 / 历史名都进别名。
- **测试方式**：改名后断言所有 `AssetProduct` / 使用记录 / 检索按旧名（经别名）与新名均可命中、计数不变；revision 记录生成。
- **PR-A 是否需处理**：**是**（PR-A 引入 family/variant 后须保证改名走 id + 别名 + revision，历史不丢；family 级别名为 PR-A 明确产物）。

### 风险 18：产品合并引用丢失（两款合并为一款后，被合并方的素材 / 使用引用消失）

- **风险**：合并若简单删表 / 改外键不留映射，则被合并产品的历史素材关系、使用次数、成片引用丢失。
- **当前状态**：【待建】。现有无产品合并操作；合并是新增需求（《业务需求》§8.2(15)、《身份血缘》§2.1"合并保留旧产品映射，引用不丢失"）。
- **架构措施**：**复用** 现有 SET NULL / id 外键引用保护风格（如 `final_video_usage.shot_id` SET NULL，《身份血缘》§2.2）；
  **新增**《通用产品目录》`ProductCatalogRevision` 记录合并映射（旧产品 → 目标产品），被合并方 id 保留并重定向、旧别名并入目标、
  使用次数与历史引用**随迁不丢、不重算为 0**（《身份血缘》§4 规则9）；产品可"合并 / 更名 / 停用"而历史关系不丢。
- **测试方式**：合并两个虚构产品后断言：被合并方素材 / 镜头 / 使用引用全部可经目标产品查到、使用次数为两者去重合并值、revision 留映射可回溯；
  按旧别名仍可检索。
- **PR-A 是否需处理**：**部分**（PR-A 建 revision / 别名基座并保证 family/variant 合并不丢引用；完整合并工作流可随《通用产品目录》落地评审）。

### 风险 19：模型更新覆盖人工结果（重识别 / 模型升级把运营已确认的产品判定冲掉）

- **风险**：模型或目录版本升级后自动重识别，若覆盖人工确认结果，则运营心血被抹、数据不可信、不敢升级模型。
- **当前状态**：【已规避（规格层）】。《身份血缘》§1.5 冻结："识别结果绑定 `model_version` + `catalog_revision`；模型 / 目录版本变化可触发重识别，
  但**人工已确认结果保留**（与 `ShotReviewState` stale 机制一致，进复核而非静默覆盖）"。现有 `apply_review()` 已保证"AI 结果不覆盖人工"（《现状差距分析》§1.4）。
- **架构措施**：**复用** 现有 `ShotReviewState` 人工优先 + `stale_*` 标记 + append-only `ReviewEvent`（人工确认优先级最高，AI 不覆盖人工）；
  **新增** 识别结果带 `model_version` / `catalog_revision`；重识别只对**未人工确认**的结果更新，对**已确认**的仅标 stale 进复核队列，
  **绝不静默覆盖**。识别结果证据分层（AI 推断 needs_human）。
- **测试方式**：先人工确认某镜头产品 → 升级 `model_version` 触发重识别 → 断言人工结果保留、仅进复核 / 标 stale、未被改写；
  未确认结果可被新模型更新；`ReviewEvent` 审计完整。
- **PR-A 是否需处理**：**部分**（PR-A 让识别结果可承载 `model_version`/`catalog_revision` 字段并复用 stale 机制；
  重识别触发与 confusable 不自动断言的执行在 PR-F）。

### 风险 20：不同产品共用同一使用阈值是否合理（一刀切阈值是否对所有品类成立）

- **风险**：`overused`（过度使用）/ `recently_used`（近期）/ "少量使用"等阈值若全局一刀切，可能对高频快消素材偏松、对稀缺精品素材偏严，
  误判使用状态、误导降权排序。
- **当前状态**：【架构规避 / 待评测确认】。《业务需求》§5、《身份血缘》§4 已冻结"使用频次 7 态阈值**全部配置化、不在规格冻结**，
  且**可按 Category 差异化**"；同时强调使用次数 / 成片引用**与产品类型无关**（计数机制通用），差异仅在**阈值取值**。
- **架构措施**：**复用** 现有"权重 / 阈值可配置、不写死"原则（检索 / 分镜均不冻结权重）；**新增** 使用阈值（低 / 高 / 过度 / 近期 N 天）
  支持**全局默认 + 按 Category 覆盖**的配置层（配置数据，非代码常量），既给通用默认、又允许品类差异化；
  **绝不**为某 seed 产品写排序 / 阈值特例。是否真需要差异化由评测决定，默认单一阈值、按需开 Category 覆盖。
- **测试方式**：同一使用数据在"全局阈值"与"Category 覆盖阈值"下派生不同 7 态的配置化测试；断言阈值来自配置而非常量、无产品特例分支；
  《评测计划》B2/B4 的"未使用优先 / 高频降权"在不同品类下分别评估阈值合理性。
- **PR-A 是否需处理**：**否（阈值属 PR-E/PR-B 使用域）**；PR-A 不涉及使用阈值。本项作为**待评测确认的开放问题**记录，
  默认通用阈值，保留按 Category 差异化的配置出口。

## 2. 审查结论汇总

### 2.1 按"当前状态"分布

| 状态 | 风险项 | 含义 |
|---|---|---|
| **已规避（结构 / 规格层已根除）** | 1, 2, 3, 9, 11, 12, 17, 19 | 现有主干数据驱动 + 规格已冻结开放集 / id 外键 / 人工优先；主要是"守住边界" |
| **部分（结构倾向规避，需 PR 补齐 / 守门）** | 4, 6, 7, 10, 13, 14, 15, 18, 20 | 已有正确倾向但需新增层级 / 入驻 / 混淆组 / 评测纪律落地 |
| **待建（本批新增需求，结构尚不存在）** | 5, 8, 16 | 动态属性体系 / 自助入驻 / 识别 Prompt 注入是新增能力 |

> 关键：**无任何一项处于"现有运行时代码已硬编码 seed 产品 / 需先清理"的状态**——硬编码审计确认 seed 名仅在 `scripts/discovery/` 审计工具，
