## 0. 项目语境与不做清单（约束 AI 能力边界）

ClipMind 是面向公司 NAS 部署的「AI 视频素材管理与智能匹配系统」，AI 仅用于**理解与匹配既有素材**：

- 只读索引视频素材；
- FFmpeg 拆镜头与派生文件（关键帧 / 缩略图 / 代理视频 / 片段裁剪——这些都是用 FFmpeg **提取 / 裁剪 / 转码**派生文件，**不是生成式能力**）；
- 外部 AI 理解打标；
- 自然语言检索；
- 画面 / 脚本匹配；
- 剪辑清单导出。

**AI 能力明确不做（排除项，任何 Provider 实现都不得越界）：**

- 文生视频
- 图生视频
- 数字人
- 声音克隆
- 视频复刻
- 替换人物 / 产品
- 自动成片

> 因此本系统的 AI Provider 接口**只承担"理解（understanding）/ 嵌入（embedding）/ 重排（rerank）/ 解析（parse）"类只读能力**，不提供任何生成式视频 / 音频能力。接口设计与实现都不得引入上述排除项。

## 1. 适配器模式与 `AIProvider` 接口

### 1.1 为什么用适配器模式

AI 服务（小米 MiMo、或未来其它 OpenAI 兼容服务）的能力、协议、配额、错误码各不相同。系统业务层（拆镜头分析、搜索解析、脚本匹配）不应直接耦合任何一家厂商。

采用**适配器模式（Adapter Pattern）**：

- 业务层只依赖统一抽象接口 `AIProvider`；
- 每个具体厂商实现一个适配器（如 `MiMoProvider`），把厂商协议翻译成统一接口；
- 通过环境变量 `AI_PROVIDER` 选择具体实现，由工厂函数装配；
- **未配置时返回 `NotConfiguredProvider`**：所有方法显式抛出"未配置"错误，绝不静默返回假数据。

这样换厂商、加厂商、降级，业务层零改动。

### 1.2 `AIProvider` 接口（7 个方法）

> 以下方法签名为设计约定，PR-01 仅在 `clipmind_shared/ai/provider.py` 中以抽象基类 + `NotConfiguredProvider` 形式落地，**不做真实实现**。

| 方法 | 用途 | 输入（要点） | 输出（要点） | 关联 PR |
| --- | --- | --- | --- | --- |
| `health_check()` | 探测连通性 / 鉴权 / 基本可用性 | 无 | 健康状态 + 探测到的 `ProviderCapabilities` | PR-03 |
| `analyze_frames(frames, ...)` | 多帧（关键帧）画面理解打标 | 关键帧图片列表（受 `AI_MAX_IMAGES` 约束） | 结构化分析结果（JSON） | PR-03 |
| `analyze_video_clip(clip, ...)` | 视频片段理解（仅当 `supports_video`） | 代理 / 片段视频 | 结构化分析结果（JSON） | PR-03 |
| `parse_search_query(text)` | 把自然语言检索词解析为结构化检索意图 | 用户检索文本 | 结构化检索意图（JSON） | PR-04 |
| `parse_script(text)` | 把脚本拆为可匹配的结构化分镜 / 需求 | 脚本文本 | 结构化分镜需求（JSON） | PR-05 |
| `generate_embedding(text\|image)` | 生成向量嵌入用于相似检索 | 文本（或图片，视能力） | 向量 | PR-04 |
| `rerank_candidates(query, candidates)` | 对候选结果重排序 | 查询 + 候选列表 | 重排后的候选 + 分数 | PR-04 / PR-05 |

> 设计要点：
> - 所有"理解 / 解析"类方法**必须返回结构化 JSON**（见第 4 节）。
> - 涉及图片 / 视频的方法在调用前**必须先依据 `ProviderCapabilities` 判定能力**，不支持则进入降级（见第 3 节），不得伪造成功。
> - `generate_embedding` 在 `supports_embeddings=false` 时不可用，须由独立 `EMBEDDING_PROVIDER` 承担（见第 3 节）。

### 1.3 `ProviderCapabilities` 字段

由 `health_check()` / 能力探测填充，作为后续所有调用与降级的**事实依据**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `supports_images` | bool | 是否支持图片输入（多帧画面理解的前提） |
| `supports_video` | bool | 是否支持视频片段输入 |
| `supports_structured_output` | bool | 是否支持稳定的结构化 JSON 输出 |
| `supports_embeddings` | bool | 是否支持向量嵌入生成 |
| `max_images_per_call` | int | 单次调用最大图片数（约束关键帧批量） |
| `context_window` | int | 上下文窗口大小（约束单次拼接长度） |

> 探测得到的 `ProviderCapabilities` 应被记录 / 缓存，作为 PR-03 之后所有 AI 调用路径的开关来源。能力未知时一律按"不支持"保守处理。

## 2. 完整能力探测清单（17 项，PR-03 执行）

> 探测**只在 PR-03 才执行真实调用**。探测脚本 / 任务应可重复运行，输出一份能力报告并据此回填 `ProviderCapabilities`。
> 探测过程必须保留**脱敏日志**（不输出完整密钥、不输出敏感原文）。

| # | 探测项 | 目的 | 影响的能力 / 行为 |
| --- | --- | --- | --- |
| 1 | 连通性 | 网络可达、端点正确 | 决定 `health_check` 是否通过 |
| 2 | 鉴权 | API Key 有效 | 决定能否调用 |
| 3 | OpenAI 兼容 | 是否兼容 OpenAI 协议格式 | 决定适配器复用方式 |
| 4 | 文本能力 | 纯文本理解 / 生成是否可用 | 文本类方法可用性（`parse_*` 兜底） |
| 5 | 严格 JSON | 能否稳定输出合法 JSON | `supports_structured_output` |
| 6 | JSON Schema | 能否按指定 Schema 约束输出 | 结构化校验策略（见第 4 节） |
| 7 | 单图理解 | 单张图片输入是否支持 | `supports_images`（基础） |
| 8 | 多图理解 | 多张图片同时输入是否支持 | `supports_images` + `max_images_per_call` |
| 9 | Base64 图片 | 是否接受 Base64 内联图片 | 关键帧传输方式选择 |
| 10 | URL 图片 | 是否接受图片 URL | 关键帧传输方式选择 |
| 11 | Embedding | 是否支持向量嵌入 | `supports_embeddings`（否则启用 `EMBEDDING_PROVIDER`） |
| 12 | 上下文窗口 | 实际可用上下文长度 | `context_window`（约束拼接量） |
| 13 | 并发 | 可并发请求数 | 批处理并发上限 |
| 14 | 超时 | 实际响应时延 / 超时阈值 | `AI_TIMEOUT` 取值依据 |
| 15 | 限流 | 速率限制策略（RPM/TPM 等） | 退避 / 排队策略 |
| 16 | 错误码 | 各类错误返回格式与语义 | 重试 / 入人工队列判定 |
| 17 | 数据是否被保留 | 厂商是否留存 / 训练我方数据 | 合规与脱敏策略、是否允许上传 |

> 探测产物：一份可读的能力探测报告 + 回填后的 `ProviderCapabilities`。任何一项探测失败，相关能力**默认置为不支持**并进入对应降级路径，而不是假设可用。

## 3. 据探测结果的降级策略

降级以第 2 节探测结果 / `ProviderCapabilities` 为唯一依据。**核心原则：宁可降级 + 标记待人工确认，绝不伪造成功。**

### 3.1 不支持图片（`supports_images = false`）

- AI 调用**降级为仅文本角色**：只用文件名 / 路径 / 时长 / 分辨率 / 已有元数据等文本信息做有限理解，不发送图片。
- **视觉镜头（依赖画面理解的分析）降级为"待人工确认"**：在审核队列中明确标记"未做视觉分析，需人工确认"。
- **绝不伪造视觉成功**：不得编造画面描述、不得把文本推测冒充为画面识别结果、不得把这类结果标记为"已完成视觉分析"。

### 3.2 不支持 Embedding（`supports_embeddings = false`）

- 不在主 Provider 上做嵌入，改由**独立的 `EMBEDDING_PROVIDER`**（独立环境变量族 `EMBEDDING_*`，见第 5 节）承担向量生成。
- 若 `EMBEDDING_PROVIDER` 也未配置，则相关检索能力（PR-04 的向量检索）显式不可用，而非返回空向量或假向量。

### 3.3 通用降级原则

- 任一能力探测为"未知"时按"不支持"保守处理。
- 任何降级都必须在结果 / 日志 / 审核队列中**留痕**，使下游与人工清楚知道"哪些是真分析、哪些是降级"。

## 4. 结构化输出约束

所有"理解 / 解析"类调用都属于机器消费，必须强约束输出。

1. **必须结构化 JSON**：所有理解 / 解析方法的输出必须是结构化 JSON，禁止自由文本作为最终结果。
2. **JSON Schema 校验**：每个方法定义对应 JSON Schema，输出必须通过 Schema 校验后才进入业务流程。
3. **失败重试**：JSON 解析失败 / Schema 校验失败时按 `AI_RETRIES` 进行有限次重试（可携带"请严格按 Schema 输出"的修正提示）。
4. **多次失败入人工队列**：重试仍失败的样本进入**人工审核队列**，标记为"AI 结构化输出失败，待人工处理"，**不得跳过、不得伪造结果**。
5. **保留脱敏日志**：每次调用与每次失败都记录**脱敏日志**（保留足以排障的元信息：方法、模型、耗时、错误码、重试次数；**不记录完整密钥、不记录敏感原文**）。

> 当探测显示 `supports_structured_output = false` 或不支持 JSON Schema 时，应优先选择支持结构化输出的模型 / 配置；在确实无法满足时，相关 AI 分析能力按降级处理并入人工队列，而非放宽校验放行脏数据。

## 5. 成本控制

外部 AI 调用是主要成本来源，必须在设计上系统性约束。

### 5.1 调用记录字段（每次 AI 调用落库）

- 调用时间（UTC，timezone-aware）
- Provider / 模型
- 方法名（如 `analyze_frames`）
- 关联实体（asset / shot / scan 等 id）
- 输入规模（图片数 / token 估算）
- 输出 token / 用量
- 估算成本
- 耗时
- 状态（成功 / 重试 / 失败 / 降级）
- 错误码（如有）

> 该记录用于额度统计、成本归因与排障。日志保持脱敏（见第 6 节）。

### 5.2 额度与节流

- **每日 / 每月额度上限**：达到上限即停止发起新的高成本调用，转入排队 / 提示，而非无上限消耗。
- **批处理**：在 `max_images_per_call`、并发上限与限流约束内合并请求，减少调用次数。

### 5.3 缓存与去重

- **结果缓存**：相同输入（如同一关键帧集合 + 同一分析意图）命中缓存直接复用。
- **已分析不重复**：对已成功分析过的素材 / 镜头不重复发起分析（除非显式 rescan / 重新分析）。
- **仅关键帧**：画面理解只针对 FFmpeg 提取出的**关键帧**，不逐帧上传整段视频。
- **人工选择后再高成本分析**：高成本分析（如视频片段级理解）放在**人工挑选 / 确认之后**触发，避免对全量素材盲目跑高成本路径。

## 6. 环境变量约定

> PR-01 仅约定键名并提供 `.env.example`（占位空值），**不写入任何真实值**。真实值在部署环境的本地环境变量 / `.env` 中提供，PR-03 起读取。

### 6.1 主 AI Provider

| 变量 | 含义 |
| --- | --- |
| `AI_PROVIDER` | 选择的 Provider 标识（如 `mimo`；未设置时使用 `NotConfiguredProvider`） |
| `AI_BASE_URL` | API 基础地址（OpenAI 兼容端点等） |
| `AI_API_KEY` | API 密钥（仅来自本地环境变量 / `.env`，绝不入库） |
| `AI_MODEL` | 使用的模型名 |
| `AI_MAX_IMAGES` | 单次调用最大图片数（不应超过探测得到的 `max_images_per_call`） |
| `AI_TIMEOUT` | 单次调用超时（秒） |
| `AI_RETRIES` | 结构化输出 / 调用失败的重试次数 |

### 6.2 独立 Embedding Provider（`supports_embeddings=false` 时启用）

| 变量 | 含义 |
| --- | --- |
| `EMBEDDING_PROVIDER` | 独立嵌入服务标识 |
| `EMBEDDING_BASE_URL` | 嵌入服务基础地址 |
| `EMBEDDING_API_KEY` | 嵌入服务密钥（同样仅本地环境变量 / `.env`） |
| `EMBEDDING_MODEL` | 嵌入模型名 |
| `EMBEDDING_TIMEOUT` | 嵌入调用超时（秒） |
| `EMBEDDING_RETRIES` | 嵌入调用重试次数 |

> 嵌入相关变量按 `EMBEDDING_*` 族独立存在，使主 Provider 不支持嵌入时可单独装配独立服务。

## 7. 密钥安全（最高优先级）

- **仅本地环境变量 / `.env` 提供密钥**：`AI_API_KEY`、`EMBEDDING_API_KEY` 等只从运行环境读取。
- **绝不入库**：密钥不写入数据库、不写入调用记录、不写入任何业务表。
- **绝不提交**：`.env` 已被 git 忽略，仓库只提交 `.env.example`（占位、空值）；**不提交任何真实密钥**。
- **日志不输出完整密钥**：所有日志（含探测日志、调用记录、错误日志）一律脱敏，最多保留可识别的尾段或哈希片段用于排障，**绝不打印完整密钥、Cookie、Token**。
- 与项目整体安全约定一致：API / web 默认仅绑 `127.0.0.1`、无鉴权不公开，AI 调用相关配置同样不对外暴露。
