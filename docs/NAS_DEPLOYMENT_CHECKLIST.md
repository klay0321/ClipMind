# NAS 部署检查清单

> 本文件用于指导将 ClipMind 部署到公司 NAS 的**正式部署前准备与执行**。
> 阅读本文件前，请先理解 ClipMind 的定位与安全红线（见 `docs/PRODUCT_REQUIREMENTS.md`）。

---

## 0. 当前状态声明（务必先读）

**当前 ClipMind 仅在 Windows 11 开发机 + Docker（Linux 容器）环境下，使用本地模拟目录 `sample_media/` 完成功能验证。**

- 尚未在任何真实 NAS 设备上完成部署或验证。
- **不得对任何人宣称"已完成真实 NAS 部署"或"已在 NAS 上跑通"。**
- 本地验证用的"源目录"是项目根的 `sample_media/`（通过 `HOST_SOURCE_DIR` 指向），它只是**模拟** NAS 的只读素材目录，不代表真实 NAS 环境。
- 真实 NAS 的 CPU 架构、操作系统、Docker 可用性、目录权限、网络策略与开发机均可能不同；下文第 1 节的检查项**全部确认通过之前，不要进入第 2 节执行部署**。

本文件描述的是"为了将来能部署而需要先确认/执行的事项"，**不是对已完成部署的记录**。

---

## 1. 正式部署前必须确认的事项（逐项核对）

下列每一项都必须在正式部署前**人工确认并记录结论**。任何一项不满足或存疑，都应暂停部署并先解决。

### 1.1 硬件与平台

- [ ] **NAS 品牌 / 型号**：记录确切品牌与型号（例如 Synology DSxxx、QNAP TS-xxx、群晖/威联通/自建 x86 服务器等）。
- [ ] **CPU 架构**：确认是 `x86_64`（amd64）还是 `ARM`（aarch64/armv7）。
  - 这是**镜像兼容性**的决定性因素，必须在第一步确认。
  - 可在 NAS 上通过 `uname -m` 查看：`x86_64` 表示 amd64；`aarch64` / `arm64` 表示 ARM64；`armv7l` 等表示 32 位 ARM。
- [ ] **内存（RAM）**：记录总内存与当前可用余量。视频派生处理（FFmpeg）与数据库对内存敏感，余量过低会导致 OOM。
- [ ] **CPU 核数与负载**：PR-02 的拆镜头与转码（FFmpeg / PySceneDetect）为 **CPU 密集**型，由专用 `media-worker` 承担。其并发默认 1（env `MEDIA_WORKER_CONCURRENCY`），可按 NAS 核数余量谨慎调高；并发越高 CPU 占用越大，需为其他服务（数据库 / api）留出余量。
- [ ] **磁盘余量**：记录系统盘与数据盘各自可用空间。需为以下用途预留：数据库（pgdata 卷）、**派生文件（关键帧 / 缩略图 / 代理视频 / 导出片段，clipmind-data 卷）**、Docker 镜像与构建缓存、日志。派生文件随拆镜头规模增长，**建议为 `/app/data` 预留约源素材容量 0.5~1 倍的空间**（实际占用取决于代理码率、镜头数与导出量）。

### 1.2 运行时与依赖

- [ ] **是否允许在该 NAS 上安装并运行 Docker / Docker Compose**：部分 NAS 固件不开放、需特定套件、或对容器有限制。确认能安装且版本满足要求。
- [ ] **PostgreSQL / Redis / FFmpeg 的获取方式**：本项目通过**容器镜像**提供 PostgreSQL（`pgvector/pgvector:pg16`）、Redis（`redis:7-alpine`），FFmpeg/FFprobe 打包在 worker 镜像内。无需在 NAS 宿主机另装这三者；但需确认 NAS 能拉取/构建这些镜像（见 1.3）。
- [ ] **目录权限**：确认运行容器的用户对 NAS 上的目标目录有正确权限：
  - 对**源素材目录**：容器只需**只读**读取权限（项目以 `:ro` 挂载并仅 `open(rb)`）。
  - 对**派生数据目录 / 命名卷**：需读写权限。
  - 注意 NAS 上的 UID/GID 映射，避免容器内进程无权读取源目录。

### 1.3 镜像架构兼容性

- [ ] **确认所有镜像在目标 CPU 架构下可用/可构建**：
  - 第三方镜像（`pgvector/pgvector:pg16`、`redis:7-alpine`）需提供目标架构（amd64 / arm64）的镜像层；若 NAS 为 ARM，务必确认对应 tag 存在。
  - 自建镜像（api / worker / web，见 `infra/*.Dockerfile`）需在**目标架构**上构建，或使用多架构构建（如 buildx）产出对应架构镜像。
  - 切勿把开发机（x86_64）构建出的 amd64 镜像直接拿到 ARM NAS 上运行。

### 1.4 网络

- [ ] **内网 or 需外网**：明确该 NAS 的网络定位。
  - 本项目 api / web **默认仅绑定 `127.0.0.1`**，无鉴权、不对公网开放。
  - 若需在内网其他机器访问，需要**显式**评估并调整访问方式（例如经由内网反向代理 + 鉴权），不可直接对公网暴露。
- [ ] **是否需要外网访问能力**：PR-03 起的 AI 理解会调用外部 AI API，需确认 NAS 能访问对应外部地址（出站）。**当前 PR-01 / PR-02 不需要外网，也不调用任何 AI**（拆镜头与派生处理全部在本机 FFmpeg 完成）。

### 1.5 数据与目录

- [ ] **需索引的实际目录**：列出 NAS 上真正要索引的源素材目录绝对路径，以及**排除规则**（要排除的子目录 / 文件名模式 / 扩展名）。
- [ ] **原始素材严格只读确认**：从制度与挂载两方面确认原始素材目录**只读**——容器以 `:ro` 挂载，应用绝不修改/删除/移动/改名/覆盖源视频，也无任何"删除源文件"的 API。
- [ ] **素材规模评估**：记录素材**数量**、**总容量**、**预计日增量**。用于评估扫描耗时、磁盘增长（派生文件）、与数据库规模。
- [ ] **AI API 文档与额度**（PR-03 起需要）：确认所选 AI Provider（如小米 MiMo）的接口文档、鉴权方式、调用额度/限流/计费。PR-01 / PR-02 不需要。
- [ ] **是否允许关键帧外发**（PR-03 起需要）：明确公司合规要求——是否允许把视频**关键帧/截图**发送到外部 AI API。若不允许，需在 PR-03 调整为不外发图片的降级策略（视觉镜头降级待人工确认）。**PR-02 仅在本机 `/app/data` 生成关键帧 / 缩略图 / 代理 / 导出片段，不外发任何图片或视频。**
- [ ] **备份目录与保留周期**：确定数据库与派生数据的备份目标目录、备份频率与保留周期。**仅备份系统派生数据与数据库，不需要也不应"备份"或写入原始素材目录。**

### 1.6 范围确认（明确不做）

部署前需与使用方确认：ClipMind **只做**只读索引、FFmpeg 拆镜头与派生文件、外部 AI 理解打标、自然语言检索、画面/脚本匹配、剪辑清单导出。

**明确不做（不得宣称具备）**：文生视频、图生视频、数字人、声音克隆、视频复刻、替换人物/产品、自动成片。

> 说明："生成关键帧/缩略图/代理视频/片段"指用 FFmpeg 从**已有素材**中**提取/裁剪/转码**派生文件，**不是**生成式 AI 能力。

## 2. 部署步骤模板（确认第 1 节全部通过后再执行）

> 以下为**模板**。真实路径以第 1.5 节确认的 NAS 目录为准。示例中 `/nas/source` 为 NAS 上的原始素材目录，`/nas/clipmind-data` 为 NAS 上的系统派生数据目录。

### 2.1 准备挂载

- **源素材目录：只读挂载**

  ```
  /nas/source:/app/source:ro
  ```

  - `:ro` 必须保留。容器内逻辑路径固定为 `/app/source`，且必须在白名单根 `ALLOWED_SOURCE_ROOTS=/app/source` 之下。
  - 本项目 compose 通过环境变量 `HOST_SOURCE_DIR` 指定宿主机源目录（本地默认 `./sample_media`）。NAS 上将其设为 `/nas/source`：

    ```env
    HOST_SOURCE_DIR=/nas/source
    ```

- **系统派生数据目录：可读写**

  ```
  /nas/clipmind-data:/app/data:rw
  ```

  - 派生文件（缩略图 / 代理视频 / 片段 / 导出 / 日志）写入此处。
  - 当前 compose 默认使用 Docker **命名卷** `clipmind-data` 挂载到 `/app/data`。若需改为 NAS 实际目录（如 `/nas/clipmind-data`）做绑定挂载，请在部署时按上面的 `:rw` 形式调整 compose 中 api/worker 的 data 挂载，并确保该目录可写。
  - 数据库数据使用命名卷 `pgdata` 持久化。

### 2.2 配置 `.env`

1. 复制示例：

   ```bash
   cp .env.example .env
   ```

2. 按 NAS 环境修改关键项（仅列与部署强相关者）：

   ```env
   # 宿主机只读源目录（指向 NAS 原始素材目录）
   HOST_SOURCE_DIR=/nas/source

   # 数据库密码：生产环境务必改为强密码（不要用默认值）
   POSTGRES_PASSWORD=<改为强密码>

   # 容器逻辑路径与白名单（一般保持默认）
   ALLOWED_SOURCE_ROOTS=/app/source
   SOURCE_MOUNT_PATH=/app/source
   DATA_DIR=/app/data
   ```

3. **`.env` 已被 git 忽略，禁止把真实密码 / 密钥 / NAS 凭据提交到仓库。** 只提交 `.env.example`。

### 2.3 启动

在项目根执行：

```bash
docker compose up -d
```

启动顺序由 compose 控制：`postgres` / `redis` 健康后，一次性 `migrate`（`alembic upgrade head`，禁止删库重建）执行成功，随后启动 `api` / `worker` / `media-worker` / `web`。

> 当前（PR-02）启用 7 个服务：`postgres`、`redis`、`migrate`、`api`、`worker`、`media-worker`、`web`。其中 `media-worker` 专用消费 `media` 队列（拆镜头 / 派生 / 导出）。**不启动** scheduler/beat、ai/export worker。

### 2.4 健康检查

- 容器级：`docker compose ps` 确认各服务为 `running`（migrate 应为已成功退出 `exited (0)`）。
- 进程存活探针（api 容器内 / 本机）：

  ```bash
  curl -fsS http://127.0.0.1:8000/health/live
  ```

- 依赖就绪探针（数据库 / Redis 等就绪）：

  ```bash
  curl -fsS http://127.0.0.1:8000/health/ready
  ```

- 系统状态：

  ```bash
  curl -fsS http://127.0.0.1:8000/api/system/status
  ```

- Web 访问：浏览器打开 `http://127.0.0.1:3000`（web 服务端把 `/api` 同源代理到内部 `api:8000`）。

### 2.5 开机自启

- compose 中各常驻服务已配置 `restart: unless-stopped`，宿主机重启 / Docker 重启后会自动拉起（migrate 为一次性任务，`restart: "no"`，符合预期）。
- 额外确认 NAS 上的 **Docker 守护进程本身设置为开机自启**，否则容器不会随 NAS 重启自动运行。

## 3. 安全红线（最高优先级，不可违反）

- **源目录只读**：源素材目录必须以 `:ro` 挂载，应用仅 `open(rb)` 读取。
- **绝不动源文件**：绝不修改 / 删除 / 移动 / 改名 / 覆盖源视频；系统**没有**任何删除源文件的 API。
- **白名单 + 逃逸防护**：源目录逻辑路径必须位于白名单根 `ALLOWED_SOURCE_ROOTS=/app/source` 之下，配合 `os.path.realpath` 包含检查与软链逃逸防护。
- **不写凭据进仓库**：禁止把 NAS 凭据、数据库密码、AI API Key、Cookie/Token 写入代码或文档。`.env` 被 git 忽略，仓库只保留 `.env.example`。
- **默认不对公网开放**：api / web 默认仅绑定 `127.0.0.1`，PR-01 无鉴权；未做鉴权与防护前不得暴露到公网。
- **关键帧外发须经授权**：在合规明确允许前，不得把关键帧 / 截图发送到外部 AI；不允许时按降级策略处理，绝不伪造视觉分析成功。

## 4. 预检脚本：`scripts/nas_preflight.sh` / `scripts/nas_preflight.ps1`

为降低"到了 NAS 才发现环境不满足"的风险，提供预检脚本，用于**在目标宿主机上**快速核对环境是否具备部署条件。

### 4.1 用途

预检脚本只做**本机环境核对**，输出每项检查的结论，帮助在执行第 2 节部署前发现明显阻塞项。它检查：

- **CPU 架构**：读取本机架构（`x86_64` / `aarch64` 等），用于判断镜像架构兼容性（对应第 1.1 / 1.3 节）。
- **Docker / Docker Compose**：是否已安装、版本是否满足、守护进程是否运行（对应第 1.2 节）。
- **内存**：总量与可用余量是否达到建议下限（对应第 1.1 节）。
- **磁盘**：目标数据目录所在磁盘的可用空间（对应第 1.1 节）。
- **挂载只读校验**：在确认源目录路径后，校验其按 `:ro` 挂载/配置的意图是否正确（例如尝试向源挂载点写入应当失败），以验证只读约束（对应第 1.5 / 第 3 节）。

- `scripts/nas_preflight.sh`：用于 Linux/类 Unix 宿主机（含真实 NAS 的 shell 环境）。
- `scripts/nas_preflight.ps1`：用于 Windows 开发机本地自检。

### 4.2 明确边界

- **预检脚本不连接真实 NAS、不读取 NAS 凭据、不访问 NAS 网络共享。** 它只检查"脚本所运行的这台机器"的本地环境。
- 预检通过**不等于**已完成 NAS 部署，也不代表已在 NAS 上验证；它只是部署前的环境体检。
- 脚本不会对源素材做任何写操作；其"只读校验"的目的恰恰是确认源目录无法被写入。

> 注：本节描述的是预检脚本的**用途与边界**。脚本文件本身按需提供于 `scripts/` 目录；若尚未创建，请勿假定其已存在或已通过。

- [ ] 源目录确认以 `:ro` 只读挂载，且应用无任何写源能力。
- [ ] `.env` 已正确配置且未提交仓库，数据库密码已改为强密码。
- [ ] 镜像已为目标 CPU 架构构建/拉取成功。
- [ ] 备份目录与保留周期已落实（仅针对派生数据与数据库）。
